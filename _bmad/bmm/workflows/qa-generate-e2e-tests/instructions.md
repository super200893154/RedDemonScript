# Quinn QA - 自动化

**目标**：为已实现的代码生成自动化 API 和 E2E 测试。

**范围**：此工作流仅生成测试。它**不**执行代码审查或故事验证（使用代码审查 `CR` 进行）。

## 说明

### 步骤 0：检测测试框架

检查项目是否有现有测试框架：

- 查找 `package.json` 依赖项（playwright、jest、vitest、cypress 等）
- 检查现有测试文件以了解模式
- 使用项目已有的任何测试框架
- 如果不存在框架：
  - 分析源代码以确定项目类型（React、Vue、Node API 等）
  - 在线搜索该技术栈当前推荐的测试框架
  - 建议元框架并使用它（或询问用户确认）

### 步骤 1：识别功能

询问用户要测试什么：

- 特定功能/组件名称
- 要扫描的目录（例如 `src/components/`）
- 或自动发现代码库中的功能

### 步骤 2：生成 API 测试（如适用）

对于 API 端点/服务，生成以下测试：

- 测试状态码（200、400、404、500）
- 验证响应结构
- 覆盖正常流程 + 1-2 个错误情况
- 使用项目现有的测试框架模式

### 步骤 3：生成 E2E 测试（如存在 UI）

对于 UI 功能，生成以下测试：

- 端到端测试用户工作流
- 使用语义定位器（角色、标签、文本）
- 专注于用户交互（点击、表单填写、导航）
- 断言可见结果
- 保持测试线性简单
- 遵循项目现有的测试模式

### 步骤 4：运行测试

执行测试以验证它们通过（使用项目的测试命令）。

如果发生失败，立即修复。

### 步骤 5：创建摘要

输出 markdown 摘要：

```markdown
# 测试自动化摘要

## 生成的测试

### API 测试
- [x] tests/api/endpoint.spec.ts - 端点验证

### E2E 测试
- [x] tests/e2e/feature.spec.ts - 用户工作流

## 覆盖率
- API 端点：已覆盖 5/10
- UI 功能：已覆盖 3/8

## 下一步
- 在 CI 中运行测试
- 根据需要添加更多边界情况
```

## 保持简单

**应该做：**

- 使用标准测试框架 API
- 专注于正常流程 + 关键错误
- 编写可读、可维护的测试
- 运行测试以验证它们通过

**避免：**

- 复杂的 fixture 组合
- 过度工程化
- 不必要的抽象

**对于高级功能：**

如果项目需要：

- 基于风险的测试策略
- 测试设计规划
- 质量门禁和 NFR 评估
- 全面的覆盖率分析
- 高级测试模式和工具

→ **安装 Test Architect (TEA) 模块**：<https://bmad-code-org.github.io/bmad-method-test-architecture-enterprise/>

## 输出

将摘要保存到：`{implementation_artifacts}/tests/test-summary.md`

**完成！** 测试已生成并验证。
