- 组件/函数/变量命名
- 路由参数格式

**结构冲突：**

- 测试位置
- 组件组织方式
- 工具和辅助函数位置
- 配置文件组织
- 静态资源组织

**格式冲突：**

- API 响应包装格式
- 错误响应结构
- API 和 UI 中的日期/时间格式
- JSON 字段命名约定
- API 状态码使用

**通信冲突：**

- 事件命名约定
- 事件负载结构
- 状态更新模式
- 操作命名约定
- 日志格式和级别

**流程冲突：**

- 加载状态处理
- 错误恢复模式

### 3. 定义模式类别

#### 命名模式

**数据库命名：**

- 表命名：users、Users 还是 user？
- 列命名：user_id 还是 userId？
- 外键格式：user_id 还是 fk_user？
- 索引命名：idx_users_email 还是 users_email_index？

**API 命名：**

- REST 端点命名：/users 还是 /user？复数还是单数？
- 路由参数格式：:id 还是 {id}？
- 查询参数命名：user_id 还是 userId？
- Header 命名约定：X-Custom-Header 还是 Custom-Header？

**代码命名：**

- 组件命名：UserCard 还是 user-card？
- 文件命名：UserCard.tsx 还是 user-card.tsx？
- 函数命名：getUserData 还是 get_user_data？
- 变量命名：userId 还是 user_id？

#### 结构模式

**项目组织：**

- 测试放在哪里？**tests**/ 还是 \*.test.ts 同目录？
- 组件如何组织？按功能还是按类型？
- 共享工具放在哪里？
- 服务和仓库如何组织？

**文件结构：**

- 配置文件位置和命名
- 静态资源组织
- 文档放置
- 环境文件组织

#### 格式模式

**API 格式：**

- API 响应包装？{data: ..., error: ...} 还是直接响应？
- 错误格式？{message, code} 还是 {error: {type, detail}}？
- JSON 中的日期格式？ISO 字符串还是时间戳？
- 成功响应结构？

**数据格式：**

- JSON 字段命名：snake_case 还是 camelCase？
- 布尔值表示：true/false 还是 1/0？
- Null 处理模式
- 单项目使用数组还是对象

#### 通信模式

**事件系统：**

- 事件命名约定：user.created 还是 UserCreated？
- 事件负载结构标准
- 事件版本控制方法
- 异步事件处理模式

**状态管理：**

- 状态更新模式：不可变更新还是直接修改？
- 操作命名约定
- 选择器模式
- 状态组织原则

#### 流程模式

**错误处理：**

- 全局错误处理方法
- 错误边界模式
- 面向用户的错误消息格式
- 日志 vs 用户错误的区分

**加载状态：**

- 加载状态命名约定
- 全局 vs 局部加载状态
- 加载状态持久化
- 加载 UI 模式
