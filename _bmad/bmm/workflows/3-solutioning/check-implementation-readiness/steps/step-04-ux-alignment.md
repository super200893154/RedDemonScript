# 步骤 4：UX 对齐检查

## 步骤目标：

检查 UX 文档是否存在，并验证其与 PRD 需求和架构决策的对齐情况，确保架构同时考虑了 PRD 和 UX 需求。

## 强制执行规则（首先阅读）：

### 通用规则：

- 🛑 绝不在没有用户输入的情况下生成内容
- 📖 关键：在采取任何行动之前阅读完整的步骤文件
- 🔄 关键：当使用 'C' 加载下一步时，确保完整阅读文件
- 📋 你是协调者，而非内容生成者
- ✅ 你必须始终以配置中 `{communication_language}` 指定的 Agent 沟通风格输出

### 角色强化：

- ✅ 你是 UX 验证者，确保用户体验得到妥善处理
- ✅ UX 需求必须得到架构支持
- ✅ 如果暗示存在 UI 但缺失 UX 文档，应发出警告
- ✅ 必须记录对齐缺口

### 步骤特定规则：

- 🎯 首先检查 UX 文档是否存在
- 🚫 不要假设不需要 UX
- 💬 验证 UX、PRD 和架构之间的对齐
- 🚪 将发现添加到输出报告

## 执行协议：

- 🎯 搜索 UX 文档
- 💾 如果找到，验证对齐情况
- 📖 如果未找到，评估是否暗示需要 UX
- 🚫 禁止在未完成评估的情况下继续

## UX 对齐检查流程：

### 1. 初始化 UX 验证

"开始**UX 对齐**验证。

我将：

1. 检查 UX 文档是否存在
2. 如果存在 UX：验证与 PRD 和架构的对齐
3. 如果不存在 UX：判断是否暗示需要 UX 并记录警告"

### 2. 搜索 UX 文档

搜索模式：

- `{planning_artifacts}/*ux*.md`（完整文档）
- `{planning_artifacts}/*ux*/index.md`（分片版）
- 在其他文档中查找 UI 相关术语

### 3. 如果存在 UX 文档

#### A. UX ↔ PRD 对齐

- 检查 UX 需求是否在 PRD 中体现
- 验证 UX 中的用户旅程是否与 PRD 用例匹配
- 识别不在 PRD 中的 UX 需求

#### B. UX ↔ 架构对齐

- 验证架构支持 UX 需求
- 检查性能需求（响应性、加载时间）
- 识别架构不支持的 UI 组件

### 4. 如果不存在 UX 文档

评估是否暗示需要 UX/UI：

- PRD 是否提到用户界面？
- 是否暗示存在 web/移动端组件？
- 这是面向用户的应用程序吗？

如果暗示需要 UX 但缺失：在报告中添加警告
