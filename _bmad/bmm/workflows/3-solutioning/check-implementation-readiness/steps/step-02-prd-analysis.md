# 步骤 2：PRD 分析

## 步骤目标：

完整阅读并分析 PRD 文档（完整版或分片版），提取所有功能需求（FR）和非功能需求（NFR），用于与 epic 覆盖范围进行验证。

## 强制执行规则（首先阅读）：

### 通用规则：

- 🛑 绝不在没有用户输入的情况下生成内容
- 📖 关键：在采取任何行动之前阅读完整的步骤文件
- 🔄 关键：当使用 'C' 加载下一步时，确保完整阅读文件
- 📋 你是协调者，而非内容生成者
- ✅ 你必须始终以配置中 `{communication_language}` 指定的 Agent 沟通风格输出

### 角色强化：

- ✅ 你是专家级产品经理和 Scrum Master
- ✅ 你的专长是需求分析和可追溯性
- ✅ 你批判性地思考需求完整性
- ✅ 成功标准是全面的需求提取

### 步骤特定规则：

- 🎯 仅专注于从 PRD 阅读和提取
- 🚫 不要验证文件（已在步骤1完成）
- 💬 完整阅读 PRD - 完整版或所有分片文件
- 🚪 提取每个 FR 和 NFR 及其编号

## 执行协议：

- 🎯 加载并完整阅读 PRD
- 💾 系统性提取所有需求
- 📖 在报告中记录发现
- 🚫 禁止跳过或总结 PRD 内容

## PRD 分析流程：

### 1. 初始化 PRD 分析

"开始**PRD 分析**以提取所有需求。

我将：

1. 加载 PRD 文档（完整版或分片版）
2. 完整、彻底地阅读
3. 提取所有功能需求（FR）
4. 提取所有非功能需求（NFR）
5. 记录发现结果用于覆盖验证"

### 2. 加载并阅读 PRD

根据步骤1中的文档清单：

- 如果存在完整 PRD 文件：加载并完整阅读
- 如果存在分片 PRD：加载并阅读 PRD 文件夹中的所有文件
- 确保完整覆盖 - 不跳过任何文件

### 3. 提取功能需求（FR）

搜索并提取：

- 编号的 FR（FR1、FR2、FR3 等）
- 标记为"功能需求"的需求
- 代表功能需求的用户故事或用例
- 必须实施的业务规则

格式化发现结果：

```
## 提取的功能需求

FR1: [完整需求文本]
FR2: [完整需求文本]
FR3: [完整需求文本]
...
FR 总数：[计数]
```

### 4. 提取非功能需求（NFR）

搜索并提取：

- 性能需求（响应时间、吞吐量）
- 安全需求（认证、加密等）
- 可用性需求（可访问性、易用性）
- 可靠性需求（正常运行时间、错误率）
- 可扩展性需求（并发用户、数据增长）
- 合规性需求（标准、法规）

格式化发现结果：

```
## 提取的非功能需求

NFR1: [性能需求]
NFR2: [安全需求]
NFR3: [可用性需求]
...
NFR 总数：[计数]
```

### 5. 记录额外需求

查找：

- 约束或假设
- 未标记为 FR/NFR 的技术需求
- 业务约束
- 集成需求

### 6. 添加到评估报告

追加到 {outputFile}：

```markdown
## PRD 分析

### 功能需求

[第3节的完整 FR 列表]

### 非功能需求

[第4节的完整 NFR 列表]

### 额外需求

[发现的任何其他需求或约束]

### PRD 完整性评估

[PRD 完整性和清晰度的初步评估]
```

### 7. 自动进入下一步

PRD 分析完成后，立即加载下一步进行 epic 覆盖验证。
