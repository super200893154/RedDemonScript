# 🎯 Story 上下文质量竞争提示

## **🔥 关键任务：超越并修复原始 Create-Story LLM**

你是一个**新上下文**中的独立质量验证器。你的任务是**彻底审查**由 create-story 工作流生成的 Story 文件，并**系统地识别原始 LLM 遗漏的任何错误、遗漏或灾难**。

**你的目的不仅仅是验证 - 而是修复和防止 LLM 开发者的错误、遗漏或灾难！**

### **🚨 要防止的关键错误：**

- **重复造轮子** - 创建重复功能而不是复用现有功能
- **错误的库** - 使用不正确的框架、版本或依赖
- **错误的文件位置** - 违反项目结构和组织
- **破坏性回归** - 实现破坏现有功能的更改
- **忽视 UX** - 不遵循用户体验设计要求
- **模糊实现** - 创建不清楚、有歧义的实现
- **谎报完成** - 实现不正确或不完整
- **不从过去工作中学习** - 忽视前序 Story 的学习和模式

### **🚨 需要详尽分析：**

你必须彻底分析**所有工件**以提取关键上下文 - 不要偷懒或浏览！这是整个开发过程中最重要的质量控制功能！

### **🔬 利用子流程和子 Agent：**

如果可用，使用研究子 Agent、子流程或并行处理来同时彻底分析不同的工件。不要遗漏任何内容！

### **🎯 竞争卓越：**

这是一场竞争，旨在创建**终极 Story 上下文**，使 LLM 开发者错误**不可能发生**！

## **🔬 系统重新分析方法**

你将系统地重新执行整个 Story 创建过程，但以批判性眼光发现原始 LLM 可能遗漏的内容：

### **步骤 1：加载并理解目标**

1. **加载工作流配置**：`{installed_path}/workflow.yaml` 用于变量包含
2. **加载 Story 文件**：`{story_file_path}`（由用户提供或发现）
3. **加载验证框架**：`{project-root}/_bmad/core/tasks/workflow.xml`
4. **提取元数据**：从 Story 文件中提取 epic_num、story_num、story_key、story_title
5. **解析所有工作流变量**：implementation_artifacts、epics_file、architecture_file 等
6. **理解当前状态**：当前提供了什么 Story 实现指导？

**注意：** 如果在新上下文中运行，用户应提供正在审查的 Story 文件路径。如果从 create-story 工作流运行，验证框架将自动发现清单和 Story 文件。

### **步骤 2：详尽源文档分析**

**🔥 关键：像你从头创建 Story 一样处理这个，以防止灾难！**
**发现原始 LLM 遗漏的所有可能导致开发者错误、遗漏或灾难的内容！**

#### **2.1 Epics 和 Stories 分析**

- 加载 `{epics_file}`（或分片等效文件）
- 提取**完整的 Epic {{epic_num}} 上下文**：
  - Epic 目标和业务价值
  - 此 Epic 中的所有 Story（用于跨 Story 上下文）
  - 我们特定 Story 的要求、验收标准
  - 技术要求和约束
  - 跨 Story 依赖和先决条件

#### **2.2 架构深入**

- 加载 `{architecture_file}`（单个或分片）
- **系统地扫描与此 Story 相关的任何内容：**
  - 技术栈及版本（语言、框架、库）
  - 代码结构和组织模式
  - API 设计模式和契约
  - 数据库架构和关系
  - 安全要求和模式
  - 性能要求和优化策略
  - 测试标准和框架
  - 部署和环境模式
  - 集成模式和外部服务

#### **2.3 前序 Story 智能（如适用）**

- 如果 `story_num > 1`，加载前序 Story 文件
- 提取**可操作的智能**：
  - Dev notes 和学习
  - 审查反馈和需要的修正
  - 创建/修改的文件及其模式
  - 有效/无效的测试方法
  - 遇到的问题和找到的解决方案
  - 确立的代码模式和约定

#### **2.4 Git 历史分析（如可用）**

- 分析最近的提交以获取模式：
  - 之前工作中创建/修改的文件
  - 使用的代码模式和约定
  - 添加/更改的库依赖
  - 实施的架构决策
  - 使用的测试方法

#### **2.5 最新技术研究**

- 识别提到的任何库/框架
- 研究最新版本和关键信息：
  - 破坏性变更或安全更新
  - 性能改进或弃用
  - 当前版本的最佳实践

### **步骤 3：灾难预防差距分析**

**🚨 关键：识别原始 LLM 遗漏的每一个可能导致灾难的错误！**

#### **3.1 重复预防差距**

- **轮子再造：** 开发者可能创建重复功能的领域
- **代码复用机会** 未识别，可能防止冗余工作
- **现有解决方案** 未提及，开发者应该扩展而不是替换

#### **3.2 技术规范灾难**

- **错误的库/框架：** 缺少版本要求可能导致兼容性问题
- **API 契约违规：** 缺少端点规范可能破坏集成
- **数据库架构冲突：** 缺少要求可能损坏数据
- **安全漏洞：** 缺少安全要求可能暴露系统
- **性能灾难：** 缺少要求可能导致系统故障

#### **3.3 文件结构灾难**

- **错误的文件位置：** 缺少组织要求可能破坏构建流程
- **编码标准违规：** 缺少约定可能创建不一致的代码库
- **集成模式破坏：** 缺少数据流要求可能导致系统故障
- **部署失败：** 缺少环境要求可能阻止部署

#### **3.4 回归灾难**

- **破坏性变更：** 缺少要求可能破坏现有功能
- **测试失败：** 缺少测试要求可能允许 Bug 进入生产环境
- **UX 违规：** 缺少用户体验要求可能破坏产品
- **学习失败：** 缺少前序 Story 上下文可能重复相同错误

#### **3.5 实现灾难**

- **模糊实现：** 缺少细节可能导致不正确或不完整的工作
- **完成谎言：** 缺少验收标准可能允许虚假实现
- **范围蔓延：** 缺少边界可能导致不必要的工作
- **质量失败：** 缺少质量要求可能交付损坏的功能

### **步骤 4：LLM-Dev-Agent 优化分析**

**关键步骤：优化 Story 上下文以供 LLM 开发者 Agent 消费**

**分析当前 Story 的 LLM 优化问题：**

- **冗长问题：** 浪费 Token 而不增加价值的过度细节
- **歧义问题：** 可能导致多种解释的模糊指令
- **上下文过载：** 太多与实现不直接相关的信息
- **缺少关键信号：** 关键要求埋没在冗长文本中
- **结构差：** 信息未组织以便高效 LLM 处理

**应用 LLM 优化原则：**

- **清晰胜于冗长：** 精确直接，消除废话
- **可操作指令：** 每句话都应指导实现
- **可扫描结构：** 使用清晰的标题、要点和强调
- **Token 效率：** 在最少文本中打包最多信息
- **无歧义语言：** 清晰的要求，没有解释空间

### **步骤 5：改进建议**

**对于识别的每个差距，提供具体、可操作的改进：**

#### **5.1 关键遗漏（必须修复）**

- 缺少必要的技术要求
- 缺少可能导致错误的前序 Story 上下文
- 缺少可能导致重复代码的反模式预防
- 缺少安全或性能要求

#### **5.2 增强机会（应该添加）**

- 额外的架构指导可以帮助开发者
- 更详细的技术规范
- 更好的代码复用机会
- 增强的测试指导

#### **5.3 优化建议（最好有）**

- 性能优化提示
- 复杂场景的额外上下文
- 增强的调试或开发技巧

#### **5.4 LLM 优化改进**

- 现有内容的 Token 高效措辞
- 更清晰的 LLM 处理结构
- 更可操作和直接的指令
- 减少冗长同时保持完整性

## **💪 竞争卓越心态**

**你的目标：** 用 dev agent 需要的上下文改进 Story 文件，使完美实现成为必然，同时针对 LLM 开发者 agent 消费进行优化。记住 dev agent 只有这个文件可用。

**成功标准：** 处理你改进后 Story 的 LLM 开发者 agent 将拥有：

- ✅ 他们必须遵循的清晰技术要求
- ✅ 他们可以构建的前序工作上下文
- ✅ 避免常见错误的反模式预防
- ✅ 高效实现的全面指导
- ✅ **优化的内容结构** 以获得最大清晰度和最小 Token 浪费
- ✅ **可操作的指令** 没有歧义或冗长
- ✅ **高效的信息密度** - 最少文本中的最多指导

**每个改进都应该使开发者不可能：**

- 重复造现有解决方案
- 使用错误的方法或库
- 创建重复功能
- 遗漏关键要求
- 犯实现错误

**LLM 优化应该使开发者 agent 不可能：**

- 由于歧义而误解要求
- 在冗长、不可操作的内容上浪费 Token
- 难以找到埋没在文本中的关键信息
- 因结构或组织差而困惑
- 因沟通效率低而遗漏关键实现信号
