# 步骤 4：自检

**目标：** 在外部审查前，根据任务、测试、验收标准和模式审核已完成的工作。

---

## 可用状态

来自之前的步骤：

- `{baseline_commit}` - 工作流开始时的 Git HEAD
- `{execution_mode}` - "tech-spec" 或 "direct"
- `{tech_spec_path}` - 技术规格文件（如果为模式 A）
- `{project_context}` - 项目模式（如果存在）

---

## 自检审核

### 1. 任务完成

验证所有任务已标记完成：

- [ ] 技术规格或心智计划中的所有任务已标记 `[x]`
- [ ] 没有任务在没有记录原因的情况下被跳过
- [ ] 任何被阻塞的任务有清晰的解释

### 2. 测试通过

验证测试状态：

- [ ] 所有现有测试仍然通过
- [ ] 为新功能编写了新测试
- [ ] 没有测试警告或无原因跳过的测试

### 3. 验收标准满足

对于每个验收标准：

- [ ] 验收标准明显满足
- [ ] 可以解释实现如何满足验收标准
- [ ] 已考虑边界情况

### 4. 模式遵循

验证代码质量：

- [ ] 遵循代码库中的现有代码模式
- [ ] 遵循 project-context 规则（如果存在）
- [ ] 错误处理与代码库一致
- [ ] 没有引入明显的代码异味

---

## 更新技术规格（仅模式 A）

如果 `{execution_mode}` 为 "tech-spec"：

1. 加载 `{tech_spec_path}`
2. 将所有任务标记为 `[x]` 完成
3. 将状态更新为"实现完成"
4. 保存更改

---

## 实现摘要

呈现摘要以过渡到审查：

```
**实现完成！**

**摘要：** {已实现的内容}
**已修改文件：** {文件列表}
**测试：** {测试摘要 - 通过/新增/等}
**验收标准状态：** {全部满足 / 已记录问题}

正在进行对抗性代码审查...
```

## 成功指标

- 所有任务验证完成
- 所有测试通过
- 所有验收标准满足
- 模式已遵循
- 技术规格已更新（如果为模式 A）
- 摘要已呈现

## 失败模式

- 声称任务完成但实际未完成
- 继续前未运行测试
- 缺失验收标准验证
- 忽略模式违规
- 未更新技术规格状态（模式 A）
