# 步骤 6：解决发现

**目标：** 交互式处理对抗性审查发现，应用修复，完成技术规格。

---

## 可用状态

来自之前的步骤：

- `{baseline_commit}` - 工作流开始时的 Git HEAD
- `{execution_mode}` - "tech-spec" 或 "direct"
- `{tech_spec_path}` - 技术规格文件（如果为模式 A）
- 步骤 5 的发现表格

---

## 解决选项

呈现："您想如何处理这些发现？"

显示：

**[W] 逐个审查** - 单独讨论每个发现
**[F] 自动修复** - 自动修复归类为"真实"的问题
**[S] 跳过** - 确认并继续提交

### 菜单处理逻辑：

- 如果 W：执行下方的逐个审查部分
- 如果 F：执行下方的自动修复部分
- 如果 S：执行下方的跳过部分

### 执行规则：

- 始终在显示菜单后暂停并等待用户输入
- 只有当用户做出选择后才继续

---

## 逐个审查 [W]

按顺序处理每个发现：

1. 呈现发现及其上下文
2. 询问：**立即修复 / 跳过 / 讨论**
3. 如果修复：立即应用修复
4. 如果跳过：记录为已确认，继续
5. 如果讨论：提供更多上下文，重新询问
6. 进入下一个发现

处理完所有发现后，总结已修复/跳过的内容。

---

## 自动修复 [F]

1. 筛选发现，仅保留归类为"真实"的
2. 为每个真实发现应用修复
3. 报告已修复的内容：

```
**已应用自动修复：**
- F1：{修复描述}
- F3：{修复描述}
...

已跳过（噪音/不确定）：F2, F4
```

---

## 跳过 [S]

1. 确认所有发现已审查
2. 记录用户选择在不修复的情况下继续
3. 继续完成

---

## 更新技术规格（仅模式 A）

如果 `{execution_mode}` 为 "tech-spec"：

1. 加载 `{tech_spec_path}`
2. 将状态更新为"已完成"
3. 添加审查备注：
   ```
   ## 审查备注
   - 对抗性审查已完成
   - 发现：{count} 总计，{fixed} 已修复，{skipped} 已跳过
   - 解决方式：{逐个审查/自动修复/跳过}
   ```
4. 保存更改

---

## 完成输出

```
**审查完成。准备提交。**

**实现摘要：**
- {已实现的内容}
- 已修改文件：{count}
- 测试：{status}
- 审查发现：{X} 已处理，{Y} 已跳过

{根据 user_skill_level 解释已实现的内容}
```

---

## 工作流完成

这是最终步骤。Quick Dev 工作流现已完成。

用户可以：

- 提交更改
- 运行额外测试
- 开始新的 Quick Dev 会话

---

## 成功指标

- 向用户呈现解决选项
- 正确执行所选方式
- 干净地应用修复（如适用）
- 技术规格已更新最终状态（模式 A）
- 提供完成摘要
- 用户理解已实现的内容

## 失败模式

- 未呈现解决选项
- 自动修复"噪音"或"不确定"的发现
- 解决后未更新技术规格（模式 A）
- 没有完成摘要
- 让用户不清楚下一步
