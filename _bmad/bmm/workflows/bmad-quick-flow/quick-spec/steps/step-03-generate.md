# 步骤 3：生成实现计划

**进度：第 3 步，共 4 步** - 下一步：审查与完成

## 规则：

- 不得跳过步骤。
- 不得优化序列。
- 必须遵循确切指令。
- 不得实现任何内容 - 仅记录。
- ✅ 你必须始终使用配置中的 `{communication_language}` 以你的 Agent 交流风格输出

## 上下文：

- 需要包含已定义"概述"和"开发上下文"部分的 `{wipFile}`。
- 重点：创建解决需求差异的实现序列，使用已捕获的技术上下文。
- 输出：具有特定文件和指令的可实现任务。
- 目标：满足 `workflow.md` 中定义的**准备开发**标准。

## 指令序列

### 1. 加载当前状态

**完整读取 `{wipFile}` 并提取：**

- 所有 frontmatter 值
- 概述部分（问题、解决方案、范围）
- 开发上下文部分（模式、文件、决策）

### 2. 生成实现计划

生成具体的实现任务：

a) **任务分解**

- 每个任务应该是一个离散、可完成的工作单元
- 任务应按逻辑顺序排列（依赖项优先）
- 在每个任务中包含要修改的具体文件
- 明确说明要进行的更改

b) **任务格式**

```markdown
- [ ] 任务 N：清晰的操作描述
  - 文件：`path/to/file.ext`
  - 操作：要进行的具体更改
  - 备注：任何实现细节
```

### 3. 生成验收标准

**创建可测试的验收标准：**

每个验收标准应遵循 Given/When/Then 格式：

```markdown
- [ ] AC N：给定 [前置条件]，当 [操作] 时，则 [预期结果]
```

**确保验收标准覆盖：**

- 正常流程功能
- 错误处理
- 边界情况（如相关）
- 集成点（如相关）

### 4. 完成附加信息

**填写剩余部分：**

a) **依赖项**

- 需要的外部库或服务
- 此功能依赖的其他任务或功能
- API 或数据依赖

b) **测试策略**

- 需要的单元测试
- 需要的集成测试
- 手动测试步骤

c) **备注**

- 事前分析中的高风险项
- 已知限制
- 未来考虑事项（范围外但值得记录）

### 5. 编写完整规格

a) **用所有生成的内容更新 `{wipFile}`：**

- 确保所有模板部分已填写
- 没有剩余的占位符文本
- 所有 frontmatter 值为最新
- 将状态更新为 'review'（不是 'ready-for-dev' - 那是在步骤 4 用户审查后）

b) **更新 frontmatter：**

```yaml
---
# ... 现有值 ...
status: 'review'
stepsCompleted: [1, 2, 3]
---
```

c) **完整阅读并遵循：** `{project-root}/_bmad/bmm/workflows/bmad-quick-flow/quick-spec/steps/step-04-review.md`（步骤 4）

## 必需输出：

- 任务必须具体、可执行、按逻辑顺序排列，并包含要修改的文件。
- 验收标准必须可测试，使用 Given/When/Then 格式。
- 状态必须更新为 'review'。

## 验证清单：

- [ ] frontmatter 中已设置 `stepsCompleted: [1, 2, 3]`。
- [ ] 规格满足**准备开发**标准。
