# 步骤 4：审查与完成

**进度：第 4 步，共 4 步** - 最终步骤

## 规则：

- 不得跳过步骤。
- 不得优化序列。
- 必须遵循确切指令。
- ✅ 你必须始终使用配置中的 `{communication_language}` 以你的 Agent 交流风格输出

## 上下文：

- 需要步骤 3 的 `{wipFile}`。
- 必须呈现完整的规格内容。反复迭代直到用户满意。
- **标准：** 规格必须满足 `workflow.md` 中定义的**准备开发**标准。

## 指令序列

### 1. 加载并呈现完整规格

**完整读取 `{wipFile}` 并从 frontmatter 中提取 `slug` 以备后用。**

**向用户呈现：**

"这是您的完整技术规格。请审查："

[显示完整的规格内容 - 所有部分]

"**快速摘要：**

- {task_count} 个待实现任务
- {ac_count} 个验收标准待验证
- {files_count} 个文件待修改"

**显示审查菜单：**

显示："**选择：** [C] 继续 [E] 编辑 [Q] 提问 [A] 高级引导 [P] Party Mode"

**暂停并等待用户选择。**

#### 菜单处理逻辑：

- 如果 C：进入第 3 部分（完成规格）
- 如果 E：进入第 2 部分（处理审查反馈），然后返回此处重新显示菜单
- 如果 Q：回答问题，然后重新显示此菜单
- 如果 A：完整阅读并遵循：`{advanced_elicitation}` 处理当前规格内容，处理增强的见解，询问用户"接受改进？(y/n)"，如果接受则更新规格然后重新显示菜单，如果不接受则保留原样然后重新显示菜单
- 如果 P：完整阅读并遵循：`{party_mode_exec}` 处理当前规格内容，处理协作见解，询问用户"接受更改？(y/n)"，如果接受则更新规格然后重新显示菜单，如果不接受则保留原样然后重新显示菜单
- 如果有其他评论或查询：提供有帮助的回复然后重新显示菜单

#### 执行规则：

- 始终在显示菜单后暂停并等待用户输入
- 只有当用户选择 'C' 时才继续完成
- 执行其他菜单项后，返回此菜单

### 2. 处理审查反馈

a) **如果用户请求更改：**

- 对 `{wipFile}` 进行请求的编辑
- 重新呈现受影响的部分
- 询问是否还有更多更改
- 循环直到用户满意

b) **如果规格不符合"准备开发"标准：**

- 指出缺失/薄弱的部分（例如，不可执行的任务、缺失的验收标准）。
- 提出具体的改进建议以达到标准。
- 在用户同意后进行编辑。

c) **如果用户有问题：**

- 回答关于规格的问题
- 澄清任何令人困惑的部分
- 如需要进行澄清性编辑

### 3. 完成规格

**当用户确认规格良好且符合"准备开发"标准时：**

a) 更新 `{wipFile}` 的 frontmatter：

   ```yaml
   ---
   # ... 现有值 ...
   status: 'ready-for-dev'
   stepsCompleted: [1, 2, 3, 4]
   ---
   ```

b) **将 WIP 文件重命名为最终文件名：**
   - 使用在第 1 部分提取的 `slug`
   - 将 `{wipFile}` 重命名为 `{implementation_artifacts}/tech-spec-{slug}.md`
   - 将此存储为 `finalFile` 用于下面的菜单

### 4. 呈现最终菜单

a) **显示完成消息和菜单：**

```
**技术规格完成！**

保存至：{finalFile}

---

**下一步：**

[A] 高级引导 - 进一步完善
[R] 对抗性审查 - 对规格进行批判性审查（强烈推荐）
[B] 开始开发 - 立即开始实现（不推荐）
[D] 完成 - 退出工作流
[P] Party Mode - 在开发前获取专家反馈

---

当你对规格完全满意后（最好经过**对抗性审查**和几轮**高级引导**），建议在全新上下文中运行实现以获得最佳效果。

复制此提示以开始开发：

\`\`\`
quick-dev {finalFile}
\`\`\`

这确保开发 agent 拥有专注于实现的干净上下文。
```

b) **暂停并等待用户选择。**

#### 菜单处理逻辑：

- 如果 A：完整阅读并遵循：`{advanced_elicitation}` 处理当前规格内容，处理增强的见解，询问用户"接受改进？(y/n)"，如果接受则更新规格然后重新显示菜单，如果不接受则保留原样然后重新显示菜单
- 如果 B：读取 `{quick_dev_workflow}` 处的完整工作流文件并按照最终规格文件的指示执行（警告：全新上下文更好）
- 如果 D：退出工作流 - 显示最终确认和规格路径
- 如果 P：完整阅读并遵循：`{party_mode_exec}` 处理当前规格内容，处理协作见解，询问用户"接受更改？(y/n)"，如果接受则更新规格然后重新显示菜单，如果不接受则保留原样然后重新显示菜单
- 如果 R：执行对抗性审查（见下文）
- 如果有其他评论或查询：提供有帮助的回复然后重新显示菜单

#### 执行规则：

- 始终在显示菜单后暂停并等待用户输入
- 执行 A、P 或 R 后，返回此菜单

#### 对抗性审查 [R] 流程：

1. **调用对抗性审查任务**：
       > 使用构建的 `{finalFile}`，加载并遵循审查任务。如果可能，使用信息不对称：在单独的子 agent 或进程中仅加载此任务，只有项目的读取权限，但没有除 `{finalFile}` 以外的任何上下文。
       <invoke-task>使用 {project-root}/_bmad/core/tasks/review-adversarial-general.xml 审查 {finalFile}</invoke-task>
       > **平台回退：** 如果任务调用不可用，加载任务文件并按其指示内联执行，传递 `{finalFile}` 作为内容。
       > 任务应：审查 `{finalFile}` 并返回发现列表。

    2. **处理发现：**
       > 从任务输出中捕获发现。
       > **如果零发现：** 暂停 - 这很可疑。重新分析或请求用户指导。
       > 评估严重性（严重、高、中、低）和有效性（真实、噪音、未定）。
       > 除非明确要求，否则不要根据严重性或有效性排除发现。
       > 按严重性排序发现。
       > 为排序后的发现编号（F1、F2、F3 等）。
       > 如果 TodoWrite 或类似工具可用，将每个发现转换为 TODO，在 TODO 中包含 ID、严重性、有效性和描述；否则将发现呈现为表格，列包括：ID、严重性、有效性、描述

    3. 返回此处并重新显示菜单。

### 5. 退出工作流

**当用户选择 [D] 时：**

"**全部完成！** 您的技术规格已就绪：

`{finalFile}`

准备实现时，运行：

```
quick-dev {finalFile}
```

开始发布！"
