# 步骤 2：映射技术约束与锚点

**进度：第 2 步，共 4 步** - 下一步：生成计划

## 规则：

- 不得跳过步骤。
- 不得优化序列。
- 必须遵循确切指令。
- 尚不能生成完整规格（那是步骤 3）。
- ✅ 你必须始终使用配置中的 `{communication_language}` 以你的 Agent 交流风格输出

## 上下文：

- 需要步骤 1 中定义了"问题陈述"的 `{wipFile}`。
- 重点：将问题陈述映射到代码库中的具体锚点。
- 输出：要触及的具体文件、要扩展的类/模式、已识别的技术约束。
- 目标：为实现计划提供可实现的基础事实。

## 指令序列

### 1. 加载当前状态

**读取 `{wipFile}` 并提取：**

- 概述部分的问题陈述和范围
- 步骤 1 中收集的任何上下文

### 2. 执行调查路径

**通用代码调查：**

_在可用的子 agent/任务中隔离深度探索。仅返回提炼的摘要以防止上下文膨胀。_

a) **基于步骤 1 的快速扫描继续**

回顾步骤 1 定向扫描中发现的内容。然后询问：

"根据我的快速查看，我发现了 [文件/模式]。还有其他文件或目录我应该深入调查吗？"

b) **读取并分析代码**

对于提供的每个文件/目录：

- 完整读取文件
- 识别模式、约定、编码风格
- 记录依赖项和导入
- 查找相关的测试文件

**如果未找到相关代码（全新开始）：**

- 识别功能应存在的目标目录。
- 扫描父目录以获取架构上下文。
- 识别应使用的标准项目工具或样板代码。
- 将此记录为"确认全新开始" - 确立不存在遗留约束。


c) **记录技术上下文**

捕获并与用户确认：

- **技术栈**：语言、框架、库
- **代码模式**：架构模式、命名约定、文件结构
- **待修改/创建的文件**：需要更改的具体文件或需要创建的新文件
- **测试模式**：测试结构、使用的测试框架

d) **查找 project-context.md**

如果 `**/project-context.md` 存在且未在步骤 1 中加载：

- 现在加载
- 提取模式和约定
- 记录必须遵循的规则

### 3. 更新 WIP 文件

**更新 `{wipFile}` 的 frontmatter：**

```yaml
---
# ... 现有 frontmatter ...
stepsCompleted: [1, 2]
tech_stack: ['{captured_tech_stack}']
files_to_modify: ['{captured_files}']
code_patterns: ['{captured_patterns}']
test_patterns: ['{captured_test_patterns}']
---
```

**更新开发上下文部分：**

填写：

- 代码库模式（来自调查）
- 参考文件表（审查的文件）
- 技术决策（调查期间做出的任何决策）

**向用户报告：**

"**已收集上下文：**

- 技术栈：{tech_stack_summary}
- 待修改文件：已识别 {files_count} 个文件
- 模式：{patterns_summary}
- 测试：{test_patterns_summary}"

### 4. 呈现检查点菜单

显示："**选择：** [A] 高级引导 [P] Party Mode [C] 继续生成规格（第 3 步，共 4 步）"

**暂停并等待用户选择。**

#### 菜单处理逻辑：

- 如果 A：完整阅读并遵循：`{advanced_elicitation}` 处理当前技术规格内容，处理增强的见解，询问用户"接受改进？(y/n)"，如果接受则更新 WIP 文件然后重新显示菜单，如果不接受则保留原样然后重新显示菜单
- 如果 P：完整阅读并遵循：`{party_mode_exec}` 处理当前技术规格内容，处理协作见解，询问用户"接受更改？(y/n)"，如果接受则更新 WIP 文件然后重新显示菜单，如果不接受则保留原样然后重新显示菜单
- 如果 C：验证 frontmatter 已更新 `stepsCompleted: [1, 2]`，然后完整阅读并遵循：`{project-root}/_bmad/bmm/workflows/bmad-quick-flow/quick-spec/steps/step-03-generate.md`
- 如果有其他评论或查询：提供有帮助的回复然后重新显示菜单

#### 执行规则：

- 始终在显示菜单后暂停并等待用户输入
- 只有当用户选择 'C' 时才继续下一步
- 执行 A 或 P 后，返回此菜单
