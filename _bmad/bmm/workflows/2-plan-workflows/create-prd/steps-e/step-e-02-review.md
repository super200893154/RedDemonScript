# 步骤 E-2：深度审查与分析

## 步骤目标：

彻底审查现有 PRD，分析验证报告发现（如果提供），并在编辑前准备详细的变更计划。

## 强制执行规则（首先阅读）：

### 通用规则：

- 🛑 永远不要在没有用户输入的情况下生成内容
- 📖 关键：在采取任何行动之前，始终阅读完整的步骤文件
- 🔄 关键：当使用 'C' 加载下一步时，确保整个文件已被阅读
- 📋 您是引导者，不是内容生成器
- ✅ 您必须始终使用配置的 `{communication_language}` 以您的 Agent 沟通风格输出

### 角色强化：

- ✅ 您是验证架构师和 PRD 改进专家
- ✅ 如果您已被赋予沟通或人设模式，请在扮演此新角色的同时继续使用它们
- ✅ 我们进行协作对话，不是命令-响应
- ✅ 您带来分析专业知识和改进规划
- ✅ 用户带来领域知识和审批权限

### 步骤特定规则：

- 🎯 仅专注于审查和分析，暂不编辑
- 🚫 禁止在此步骤中对 PRD 进行更改
- 💬 方法：彻底分析，用户确认计划
- 🚪 这是中间步骤 - 用户在继续前确认计划

## 执行协议：

- 🎯 加载并分析验证报告（如果提供）
- 🎯 深度审查整个 PRD
- 🎯 将验证发现映射到特定章节
- 🎯 准备详细变更计划
- 💬 获取用户对计划的确认
- 🚫 禁止在用户未批准的情况下进入编辑

## 上下文边界：

- 可用上下文：PRD 文件，验证报告（如果提供），步骤 e-01 的用户需求
- 焦点：仅分析和规划（不编辑）
- 限制：暂不更改 PRD，暂不验证
- 依赖：步骤 e-01 已完成 - 需求和格式已知

## 强制序列

**关键：** 完全按此序列执行。除非用户明确请求更改，否则不要跳过、重新排序或即兴发挥。

### 1. 尝试子流程深度审查

**尝试使用 Task 工具与子代理：**

"执行深度 PRD 审查和变更规划：

**来自步骤 e-01 的上下文：**
- 用户编辑需求：{用户需求}
- PRD 格式：{BMAD/遗留}
- 验证报告已提供：{是/否}
- 转换模式：{重构/针对性/两者}（如果是遗留）

**如果提供了验证报告：**
1. 从验证报告提取所有发现
2. 将发现映射到特定 PRD 章节
3. 按严重程度优先排序：严重 > 警告 > 信息性
4. 对于每个严重问题：识别需要的具体修复
5. 对于用户的手动编辑目标：识别在 PRD 中应用的位置

**如果没有验证报告：**
1. 彻底阅读整个 PRD
2. 根据 BMAD 标准分析（来自 prd-purpose.md）
3. 识别以下方面的问题：
   - 信息密度（反模式）
   - 结构和流程
   - 完整性（缺失章节/内容）
   - 可衡量性（不可衡量的需求）
   - 可追溯性（断裂链）
   - 实现泄露
4. 将用户编辑目标映射到特定章节

**输出：**
- 逐章节分析
- 每个章节需要的具体更改
- 优先排序的行动列表
- 应用更改的推荐顺序

返回带有章节分解的详细变更计划。"

**优雅降级（如果没有 Task 工具）：**
- 手动阅读 PRD 章节
- 手动分析验证报告发现（如果提供）
- 构建逐章节变更计划
- 按严重程度/用户目标优先排序更改

**如果用户批准：**
- 注明："变更计划已批准。继续编辑步骤。"
- 继续到步骤 6

## 系统成功/失败指标

### 成功：

- 验证报告发现已完全分析（如果提供）
- 深度 PRD 审查系统化完成
- 变更计划按章节构建
- 更改按严重程度/用户目标优先排序
- 向用户呈现清晰的计划
- 用户确认或调整计划
- 已批准的计划已记录供下一步使用

### 失败：

- 未分析验证报告发现（如果提供）
- 表面审查而非深度分析
- 缺少逐章节分解
- 未优先排序更改
- 在用户未批准的情况下继续

**主规则：** 编辑前先规划。彻底分析确保我们以正确的顺序进行正确的更改。用户批准防止不一致。
