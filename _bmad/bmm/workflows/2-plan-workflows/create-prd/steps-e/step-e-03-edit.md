# 步骤 E-3：编辑与更新

## 步骤目标：

按照步骤 e-02 中已批准的变更计划对 PRD 应用更改，包括内容更新、结构改进和格式转换（如果需要）。

## 强制执行规则（首先阅读）：

### 通用规则：

- 🛑 始终在用户输入/批准的情况下生成内容
- 📖 关键：在采取任何行动之前，始终阅读完整的步骤文件
- 🔄 关键：当使用 'C' 加载下一步时，确保整个文件已被阅读
- 📋 您是引导者，不是内容生成器
- ✅ 您必须始终使用配置的 `{communication_language}` 以您的 Agent 沟通风格输出

### 角色强化：

- ✅ 您是验证架构师和 PRD 改进专家
- ✅ 如果您已被赋予沟通或人设模式，请在扮演此新角色的同时继续使用它们
- ✅ 我们进行协作对话，不是命令-响应
- ✅ 您带来分析专业知识和精确编辑技能
- ✅ 用户带来领域知识和审批权限

### 步骤特定规则：

- 🎯 仅专注于执行步骤 e-02 中已批准的更改
- 🚫 禁止在已批准计划之外进行更改
- 💬 方法：有条不紊，逐章节执行
- 🚪 这是中间步骤 - 用户可以请求调整

## 执行协议：

- 🎯 系统化遵循已批准的变更计划
- 💾 按计划编辑 PRD 内容
- 📖 按需更新 frontmatter
- 🚫 禁止在未完成的情况下继续

## 上下文边界：

- 可用上下文：PRD 文件，步骤 e-02 中已批准的变更计划，prd-purpose 标准
- 焦点：仅执行已批准计划中的更改
- 限制：不添加计划外的更改，暂不验证
- 依赖：步骤 e-02 已完成 - 计划已由用户批准

## 强制序列

**关键：** 完全按此序列执行。除非用户明确请求更改，否则不要跳过、重新排序或即兴发挥。

### 1. 检索已批准的变更计划

从步骤 e-02 检索：
- **已批准的更改：** 逐章节列表
- **优先顺序：** 应用更改的顺序
- **用户需求：** 步骤 e-01 的编辑目标

显示："**开始 PRD 编辑**

**变更计划：** {摘要}
**总更改数：** {计数}
**估计工作量：** {工作量级别}

**正在逐章节进行编辑...**"

### 2. 尝试子流程编辑（对于复杂更改）

**尝试使用 Task 工具与子代理处理主要章节：**

"执行 {章节名称} 的 PRD 编辑：

**上下文：**
- 要编辑的章节：{章节名称}
- 当前内容：{现有内容}
- 需要的更改：{计划中的具体更改}
- BMAD PRD 标准：从 prd-purpose.md 加载

**任务：**
1. 阅读当前 PRD 章节
2. 应用指定的更改
3. 确保 BMAD PRD 原则合规：
   - 高信息密度（无填充）
   - 可衡量的需求
   - 清晰的结构
   - 正确的 markdown 格式
4. 返回更新后的章节内容

应用更改并返回更新后的章节。"

**优雅降级（如果没有 Task 工具）：**
- 在当前上下文中直接执行编辑
- 加载 PRD 章节，应用更改，保存

### 3. 逐章节执行更改

**对于已批准计划中的每个章节（按优先顺序）：**

**a) 加载当前章节**
- 阅读当前 PRD 章节内容
- 注意存在的内容

**b) 按计划应用更改**
- 添加：用适当内容创建新章节
- 更新：按计划修改现有内容
- 删除：删除指定的内容
- 重构：将内容重新格式化为 BMAD 标准

**c) 更新 PRD 文件**
- 对 PRD 应用更改
- 保存更新后的 PRD
- 验证更改已正确应用

**每个章节后显示进度：**
"**章节已更新：** {章节名称}
更改：{简要摘要}
{还有更多章节...}"

### 4. 处理重构（如果需要）

**如果转换模式是"全面重构"或"两者"：**

**对于重构：**
- 将 PRD 重组为 BMAD 标准结构
- 确保正确的 ## 二级标题
- 逻辑重新排序章节
- 更新 PRD frontmatter 以匹配 BMAD 格式

**遵循 BMAD PRD 结构：**
1. 执行摘要
2. 成功标准
3. 产品范围
4. 用户旅程
5. 领域需求（如适用）
6. 创新分析（如适用）
7. 项目类型需求
8. 功能需求
9. 非功能需求

显示："**PRD 已重构**
已应用 BMAD 标准结构。
{添加/重新排序的章节}"

### 5. 更新 PRD Frontmatter

**确保 frontmatter 完整且准确：**

```yaml
---
workflowType: 'prd'
workflow: 'create'  # 或 'validate' 或 'edit'
classification:
  domain: '{领域}'
  projectType: '{项目类型}'
  complexity: '{复杂度}'
inputDocuments: [输入文档列表]
stepsCompleted: ['step-e-01-discovery', 'step-e-02-review', 'step-e-03-edit']
lastEdited: '{当前日期}'
editHistory:
  - date: '{当前日期}'
    changes: '{更改摘要}'
---
```

**相应更新 frontmatter。**

### 6. 最终审查更改

**加载完整更新后的 PRD**

**验证：**
- 所有已批准的更改已正确应用
- PRD 结构健全
- 无意外修改
- Frontmatter 准确

**如果发现问题：**
- 立即修复
- 注明已进行的更正

**如果用户想要调整：**
- 接受反馈并进行调整
- 调整后重新验证

### 7. 确认完成

显示：

"**PRD 编辑完成**

**已应用的更改：** {计数} 个章节已修改
**PRD 已更新：** {prd_file_path}

**更改摘要：**
{主要更改的简要列表}

**PRD 已准备好用于：**
- 下游工作流（UX、架构）
- 验证（如果尚未验证）

**您接下来想做什么？**"

### 8. 呈现菜单选项

**[V] 运行验证** - 执行完整验证工作流（steps-v/step-v-01-discovery.md）
**[S] 仅摘要** - 以更改摘要结束（不验证）
**[A] 调整** - 进行额外编辑
**[X] 退出** - 退出编辑工作流

#### 执行规则：

- 始终暂停并等待用户输入
- 仅根据用户选择继续

#### 菜单处理逻辑：

- 如果 V（验证）：显示"正在启动验证工作流..."然后完整阅读并遵循：steps-v/step-v-01-discovery.md
- 如果 S（摘要）：呈现编辑摘要并退出
- 如果 A（调整）：接受额外需求，循环回编辑
- 如果 X（退出）：显示摘要并退出
