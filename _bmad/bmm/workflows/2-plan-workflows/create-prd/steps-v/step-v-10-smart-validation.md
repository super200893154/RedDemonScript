# 步骤 10：SMART 需求验证

## 步骤目标：

验证功能需求是否符合 SMART 质量标准（具体、可衡量、可实现、相关、可追溯），确保高质量需求。

## 强制执行规则（首先阅读）：

### 通用规则：

- 🛑 永远不要在没有用户输入的情况下生成内容
- 📖 关键：在采取任何行动之前，始终阅读完整的步骤文件
- 🔄 关键：当使用 'C' 加载下一步时，确保整个文件已被阅读
- 📋 您是引导者，不是内容生成器
- ✅ 您必须始终使用配置的 `{communication_language}` 以您的 Agent 沟通风格输出

### 角色强化：

- ✅ 您是验证架构师和质量保证专员
- ✅ 如果您已被赋予沟通或人设模式，请在扮演此新角色的同时继续使用它们
- ✅ 我们进行系统化验证，不是协作对话
- ✅ 您带来需求工程专业知识和质量评估能力
- ✅ 此步骤自动运行 - 无需用户输入

### 步骤特定规则：

- 🎯 仅专注于使用 SMART 框架的 FR 质量评估
- 🚫 禁止在此步骤中验证其他方面
- 💬 方法：对每个 FR 按 SMART 标准评分（1-5 分）
- 🚪 这是验证序列步骤 - 完成后自动继续

## 执行协议：

- 🎯 从 PRD 中提取所有 FR
- 🎯 对每个 FR 按 SMART 标准评分（具体、可衡量、可实现、相关、可追溯）
- 💾 标记任何类别得分 < 3 的 FR
- 📖 将评分表和建议追加到验证报告
- 📖 显示"正在进行下一个检查..."并加载下一步
- 🚫 禁止暂停或请求用户输入

## 上下文边界：

- 焦点：仅使用 SMART 框架的 FR 质量评估

- 依赖：步骤 2-9 已完成 - 全面的验证检查完成

## 强制序列

**关键：** 完全按此序列执行。除非用户明确请求更改，否则不要跳过、重新排序或即兴发挥。

### 1. 提取所有功能需求

从 PRD 的功能需求章节中提取：
- 所有 FR 及其 FR 编号（FR-001、FR-002 等）
- 统计总 FR 数

### 2. 尝试子流程验证

**尝试使用 Task 工具启动子流程：**

"对这些功能需求执行 SMART 需求验证：

{列出所有 FR}

**对于每个 FR，按 SMART 标准评分（1-5 分）：**

**具体（1-5）：**
- 5：清晰、无歧义、定义良好
- 3：有些清晰但可以更具体
- 1：模糊、有歧义、不清晰

**可衡量（1-5）：**
- 5：可量化指标、可测试
- 3：部分可衡量
- 1：不可衡量、主观

**可实现（1-5）：**
- 5：现实、在约束条件下可实现
- 3：可能可实现但不确定
- 1：不现实、技术上不可行

**相关（1-5）：**
- 5：清晰对齐用户需求和业务目标
- 3：有些相关但联系不清晰
- 1：不相关、与目标不一致

**可追溯（1-5）：**
- 5：清晰追溯到用户旅程或业务目标
- 3：部分可追溯
- 1：孤立需求、无清晰来源

**对于任何类别得分 < 3 的每个 FR：**
- 提供具体的改进建议

返回带有所有 FR 分数和低分 FR 改进建议的评分表。"

**优雅降级（如果没有 Task 工具）：**
- 手动对每个 FR 按 SMART 标准评分
- 注意低分 FR
- 提供改进建议

### 3. 构建评分表

对于每个 FR：
- FR 编号
- 具体分数（1-5）
- 可衡量分数（1-5）
- 可实现分数（1-5）
- 相关分数（1-5）
- 可追溯分数（1-5）
- 平均分数
- 如果任何类别 < 3 则标记

**计算整体 FR 质量：**
- 所有分数 ≥ 3 的 FR 百分比
- 所有分数 ≥ 4 的 FR 百分比
- 所有 FR 和类别的平均分数

### 4. 向验证报告报告 SMART 发现

追加到验证报告：

```markdown
## SMART 需求验证

**功能需求总数：** {计数}

### 评分摘要

**所有分数 ≥ 3：** {百分比}%（{计数}/{总计}）
**所有分数 ≥ 4：** {百分比}%（{计数}/{总计}）
**整体平均分数：** {平均}/5.0

### 评分表

| FR # | 具体 | 可衡量 | 可实现 | 相关 | 可追溯 | 平均 | 标记 |
|------|------|--------|--------|------|--------|------|------|
| FR-001 | {s1} | {m1} | {a1} | {r1} | {t1} | {avg1} | {如果任何 <3 则为 X} |
| FR-002 | {s2} | {m2} | {a2} | {r2} | {t2} | {avg2} | {如果任何 <3 则为 X} |
[继续所有 FR]

**图例：** 1=差，3=可接受，5=优秀
**标记：** X = 一个或多个类别分数 < 3

### 改进建议

**低分 FR：**

**FR-{编号}：** {具体改进建议}
[对于任何类别得分 < 3 的每个 FR]

### 总体评估

**严重程度：** [如果 >30% FR 被标记为严重，10-30% 为警告，<10% 为通过]

**建议：**
[如果严重] "许多 FR 存在质量问题。使用 SMART 框架修订标记的 FR 以提高清晰度和可测试性。"
[如果警告] "一些 FR 可以从 SMART 细化中受益。重点关注上述标记的需求。"
[如果通过] "功能需求整体展现了良好的 SMART 质量。"
```

### 5. 显示进度并自动继续

显示："**SMART 需求验证完成**

FR 质量：{百分比}% 得分可接受（{严重程度}）

**正在进行下一个验证检查...**"

立即完整阅读并遵循：{nextStepFile}（step-v-11-holistic-quality-validation.md）

## 系统成功/失败指标

### 成功：

- 所有 FR 从 PRD 中提取
- 每个 FR 按所有 5 个 SMART 标准评分（1-5 分）
- 分数 < 3 的 FR 已标记改进
- 为低分 FR 提供改进建议
- 评分表已构建，包含所有 FR 分数
- 整体质量评估已计算
- 发现已报告到验证报告
- 自动进入下一个验证步骤
- 已尝试子流程并优雅降级

### 失败：

- 未按所有 SMART 标准对所有 FR 评分
- 缺少低分 FR 的改进建议
- 未构建评分表
- 未计算整体质量指标
- 未向验证报告报告发现
- 未自动继续

**主规则：** FR 应该是高质量的，而不仅仅是存在。SMART 框架提供客观的质量衡量标准。
