# 步骤 4：产品简报覆盖验证

## 步骤目标：

验证 PRD 是否覆盖产品简报的所有内容（如果简报被用作输入），将简报内容映射到 PRD 章节并识别差距。

## 强制执行规则（首先阅读）：

### 通用规则：

- 🛑 永远不要在没有用户输入的情况下生成内容
- 📖 关键：在采取任何行动之前，始终阅读完整的步骤文件
- 🔄 关键：当使用 'C' 加载下一步时，确保整个文件已被阅读
- 📋 您是引导者，不是内容生成器
- ✅ 您必须始终使用配置的 `{communication_language}` 以您的 Agent 沟通风格输出

### 角色强化：

- ✅ 您是验证架构师和质量保证专员
- ✅ 如果您已被赋予沟通或人设模式，请在扮演此新角色的同时继续使用它们
- ✅ 我们进行系统化验证，不是协作对话
- ✅ 您带来分析严谨性和可追溯性专业知识
- ✅ 此步骤自动运行 - 无需用户输入

### 步骤特定规则：

- 🎯 仅专注于产品简报覆盖（取决于简报是否存在）
- 🚫 禁止在此步骤中验证其他方面
- 💬 方法：系统化映射和差距分析
- 🚪 这是验证序列步骤 - 完成后自动继续

## 执行协议：

- 🎯 检查输入文档中是否存在产品简报
- 💬 如果无简报：跳过此检查并报告"不适用 - 无产品简报"
- 🎯 如果简报存在：将简报内容映射到 PRD 章节
- 💾 将覆盖发现追加到验证报告
- 📖 显示"正在进行下一个检查..."并加载下一步
- 🚫 禁止暂停或请求用户输入

## 上下文边界：

- 可用上下文：PRD 文件，步骤 1 的输入文档，验证报告
- 焦点：仅产品简报覆盖（条件性）
- 限制：不验证其他方面，条件执行
- 依赖：步骤 1 已完成 - 输入文档已加载

## 强制序列

**关键：** 完全按此序列执行。除非用户明确请求更改，否则不要跳过、重新排序或即兴发挥。

### 1. 检查产品简报

检查步骤 1 的 inputDocuments 中是否加载了产品简报：

**如果未找到产品简报：**
追加到验证报告：
```markdown
## 产品简报覆盖

**状态：** 不适用 - 未提供产品简报作为输入
```

显示："**产品简报覆盖：已跳过**（未提供产品简报）

**正在进行下一个验证检查...**"

立即完整阅读并遵循：{nextStepFile}

**如果产品简报存在：** 继续下面的步骤 2

### 2. 尝试子流程验证

**尝试使用 Task 工具启动子流程：**

"执行产品简报覆盖验证：

1. 加载产品简报
2. 提取关键内容：
   - 愿景陈述
   - 目标用户/画像
   - 问题陈述
   - 关键功能
   - 目标/目的
   - 差异化因素
   - 约束
3. 对于每项，在 PRD 中搜索相应覆盖
4. 分类覆盖：完全覆盖 / 部分覆盖 / 未找到 / 故意排除
5. 注意任何差距及严重程度：严重 / 中等 / 信息性

返回带有分类的结构化覆盖图。"

### 3. 优雅降级（如果 Task 工具不可用）

如果 Task 工具不可用，直接执行分析：

**从产品简报中提取：**
- 愿景：这是什么产品？
- 用户：面向谁？
- 问题：解决什么问题？
- 功能：关键能力是什么？
- 目标：成功标准是什么？
- 差异化：有什么独特之处？

**对于每项，搜索 PRD：**
- 扫描执行摘要查找愿景
- 检查用户旅程或用户画像
- 查找问题陈述
- 审查功能需求查找功能
- 检查成功标准章节
- 搜索差异化因素

**分类覆盖：**
- **完全覆盖：** 内容存在且完整
- **部分覆盖：** 内容存在但不完整
- **未找到：** PRD 中缺失内容
- **故意排除：** 内容明确在范围外

### 4. 评估覆盖和严重程度

**对于每个差距（部分覆盖或未找到）：**
- 这是严重的吗？（核心愿景、主要用户、主要功能）
- 这是中等的吗？（次要功能、一些目标）
- 这是信息性的吗？（锦上添花的功能、小细节）

**注意：** 某些排除可能是故意的（有效的范围决策）

### 5. 向验证报告报告覆盖发现

追加到验证报告：

```markdown
## 产品简报覆盖

**产品简报：** {简报文件名}

### 覆盖图

**愿景陈述：** [完全/部分/未找到/故意排除]
[如果存在差距：注意严重程度和具体缺失内容]

**目标用户：** [完全/部分/未找到/故意排除]
[如果存在差距：注意严重程度和具体缺失内容]

**问题陈述：** [完全/部分/未找到/故意排除]
[如果存在差距：注意严重程度和具体缺失内容]

**关键功能：** [完全/部分/未找到/故意排除]
[如果存在差距：列出具体功能及严重程度]

**目标/目的：** [完全/部分/未找到/故意排除]
[如果存在差距：注意严重程度和具体缺失内容]

**差异化因素：** [完全/部分/未找到/故意排除]
[如果存在差距：注意严重程度和具体缺失内容]

### 覆盖摘要

**总体覆盖：** [百分比或定性评估]
**严重差距：** [数量] [如有则列出]
**中等差距：** [数量] [如有则列出]
**信息性差距：** [数量] [如有则列出]

**建议：**
[如果存在严重差距] "PRD 应修订以覆盖关键的产品简报内容。"
[如果存在中等差距] "考虑处理中等差距以实现完整覆盖。"
[如果差距极少] "PRD 对产品简报内容提供了良好覆盖。"
```

## 系统成功/失败指标

### 成功：

- 正确检查产品简报是否存在
- 如果无简报：报告"不适用"并优雅跳过
- 如果简报存在：将所有关键简报内容映射到 PRD 章节
- 覆盖分类适当（完全/部分/未找到/故意排除）
- 差距严重程度已评估（严重/中等/信息性）
- 发现已报告到验证报告
- 自动进入下一个验证步骤
- 已尝试子流程并优雅降级

### 失败：

- 在尝试验证前未检查简报是否存在
- 如果简报存在：未映射所有关键内容区域
- 缺少覆盖分类
- 未向验证报告报告发现
- 未自动继续

**主规则：** 产品简报覆盖是条件性的 - 无简报则跳过，有简报则彻底验证。始终自动继续。
