# 步骤 5：可衡量性验证

## 步骤目标：

验证所有功能需求（FR）和非功能需求（NFR）都是可衡量、可测试的，并遵循正确格式且不包含实现细节。

- ✅ 如果您已被赋予沟通或人设模式，请在扮演此新角色的同时继续使用它们

- ✅ 您带来分析严谨性和需求工程专业知识
- ✅ 此步骤自动运行 - 无需用户输入

### 1. 尝试子流程验证

**尝试使用 Task 工具启动子流程：**

"对此 PRD 执行可衡量性验证：

**功能需求（FR）：**
1. 从功能需求章节提取所有 FR
2. 检查每个 FR：
   - '[参与者] 可以 [能力]' 格式合规性
   - 无主观形容词（简单、快速、简洁、直观等）
   - 无模糊量词（多个、若干、一些、许多等）
   - 无实现细节（技术名称、库名称、数据结构，除非与能力相关）
3. 记录违规及行号

**非功能需求（NFR）：**
1. 从非功能需求章节提取所有 NFR
2. 检查每个 NFR：
   - 带有测量方法的具体指标
   - 模板合规性（标准、指标、测量方法、上下文）
   - 包含上下文（为什么重要，影响谁）
3. 记录违规及行号

返回带有违规计数和示例的结构化发现。"

### 2. 优雅降级（如果 Task 工具不可用）

如果 Task 工具不可用，直接执行分析：

**功能需求分析：**

提取所有 FR 并检查每个：

**格式合规性：**
- 是否遵循"[参与者] 可以 [能力]"模式？
- 参与者是否清晰定义？
- 能力是否可操作和可测试？

**无主观形容词：**
- 扫描：简单、快速、简洁、直观、用户友好、响应快、快速、高效（无指标）
- 记录行号

**无模糊量词：**
- 扫描：多个、若干、一些、许多、少量、各种、一定数量
- 记录行号

**无实现细节：**
- 扫描：React、Vue、Angular、PostgreSQL、MongoDB、AWS、Docker、Kubernetes、Redux 等
- 除非与能力相关（例如，"API 消费者可以访问..."）
- 记录行号

**非功能需求分析：**

提取所有 NFR 并检查每个：

**具体指标：**
- 是否有可衡量的标准？（例如，"响应时间 < 200ms"，不是"快速响应"）
- 这是否可以测量或测试？

**模板合规性：**
- 标准是否定义？
- 指标是否指定？
- 测量方法是否包含？
- 上下文是否提供？

### 3. 统计违规

**FR 违规：**
- 格式违规：计数
- 主观形容词：计数
- 模糊量词：计数
- 实现泄露：计数
- FR 违规总计：总和

**NFR 违规：**
- 缺失指标：计数
- 模板不完整：计数
- 缺失上下文：计数
- NFR 违规总计：总和

**总违规：** FR 违规 + NFR 违规
