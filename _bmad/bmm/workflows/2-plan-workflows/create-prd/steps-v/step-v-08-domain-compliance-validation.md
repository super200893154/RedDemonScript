# 步骤 8：领域合规验证

## 步骤目标：

验证高复杂度领域（医疗健康、金融科技、政务科技等）的特定领域需求是否存在，确保监管和合规需求已正确记录。

## 强制执行规则（首先阅读）：

### 通用规则：

- 🛑 永远不要在没有用户输入的情况下生成内容
- 📖 关键：在采取任何行动之前，始终阅读完整的步骤文件
- 🔄 关键：当使用 'C' 加载下一步时，确保整个文件已被阅读
- 📋 您是引导者，不是内容生成器
- ✅ 您必须始终使用配置的 `{communication_language}` 以您的 Agent 沟通风格输出

### 角色强化：

- ✅ 您是验证架构师和质量保证专员
- ✅ 如果您已被赋予沟通或人设模式，请在扮演此新角色的同时继续使用它们
- ✅ 我们进行系统化验证，不是协作对话
- ✅ 您带来领域专业知识和合规知识
- ✅ 此步骤自动运行 - 无需用户输入

### 步骤特定规则：

- 🎯 仅专注于特定领域合规需求
- 🚫 禁止在此步骤中验证其他方面
- 💬 方法：基于领域分类的条件验证
- 🚪 这是验证序列步骤 - 完成后自动继续

## 执行协议：

- 🎯 检查 PRD frontmatter 中的 classification.domain
- 💬 如果低复杂度（通用）：跳过详细检查
- 🎯 如果高复杂度：验证必需的特殊章节
- 💾 将合规发现追加到验证报告
- 📖 显示"正在进行下一个检查..."并加载下一步
- 🚫 禁止暂停或请求用户输入

## 上下文边界：

- 可用上下文：带有 frontmatter 分类的 PRD 文件，验证报告
- 焦点：仅领域合规（取决于领域复杂度）
- 限制：不验证其他方面，条件执行
- 依赖：步骤 2-7 已完成 - 格式和需求验证完成

## 强制序列

**关键：** 完全按此序列执行。除非用户明确请求更改，否则不要跳过、重新排序或即兴发挥。

### 1. 加载领域复杂度数据

加载并完整阅读以下文件：
`{domainComplexityData}`（../data/domain-complexity.csv）

此 CSV 包含：
- 领域分类和复杂度级别（高/中/低）
- 每个领域必需的特殊章节
- 受监管行业的关键关注点和需求

内化此数据 - 它决定哪些领域需要特殊合规章节。

### 2. 提取领域分类

从 PRD frontmatter 中提取：
- `classification.domain` - 此 PRD 属于哪个领域？

**如果未找到领域分类：**
视为"通用"（低复杂度）并继续到步骤 4

### 2. 确定领域复杂度

**低复杂度领域（跳过详细检查）：**
- 通用
- 消费者应用（标准电商、社交、生产力）
- 内容网站
- 商业工具（标准）

**高复杂度领域（需要特殊章节）：**
- 医疗健康 / 健康科技
- 金融科技 / 金融服务
- 政务科技 / 公共部门
- 教育科技（教育记录、认证课程）
- 法律科技
- 其他受监管领域

### 3. 对于高复杂度领域：验证必需的特殊章节

**尝试子流程验证：**

"对 {领域} 执行领域合规验证：

基于 {领域} 需求，检查 PRD：

**医疗健康：**
- 临床需求章节
- 监管路径（FDA、HIPAA 等）
- 安全措施
- HIPAA 合规（数据隐私、安全）
- 患者安全考虑

**金融科技：**
- 合规矩阵（SOC2、PCI-DSS、GDPR 等）
- 安全架构
- 审计需求
- 欺诈预防措施
- 金融交易处理

**政务科技：**
- 无障碍标准（WCAG 2.1 AA、第 508 条）
- 采购合规
- 安全许可需求
- 数据驻留需求

**其他受监管领域：**
- 检查特定领域的监管章节
- 合规需求
- 特殊考虑

对于每个必需章节：
- PRD 中是否存在？
- 文档是否充分？
- 注意任何差距

返回带有存在/充分性评估的合规矩阵。"

**优雅降级（如果没有 Task 工具）：**
- 根据领域手动检查必需章节
- 列出存在的章节和缺失的章节
- 评估文档充分性

### 5. 对于低复杂度领域：跳过详细检查

追加到验证报告：
```markdown
## 领域合规验证

**领域：** {领域}
**复杂度：** 低（通用/标准）
**评估：** 不适用 - 无特殊领域合规需求

**注意：** 此 PRD 用于标准领域，无监管合规需求。
```

显示："**领域合规验证已跳过**

领域：{领域}（低复杂度）

**正在进行下一个验证检查...**"

立即完整阅读并遵循：{nextStepFile}

### 6. 报告合规发现（高复杂度领域）

追加到验证报告：

```markdown
## 领域合规验证

**领域：** {领域}
**复杂度：** 高（受监管）

### 必需的特殊章节

**{章节 1 名称}：** [存在/缺失/充分]
{如果缺失或不充分：注意具体差距}

**{章节 2 名称}：** [存在/缺失/充分]
{如果缺失或不充分：注意具体差距}

[继续所有必需章节]

### 合规矩阵

| 需求 | 状态 | 备注 |
|------|------|------|
| {需求 1} | [满足/部分/缺失] | {备注} |
| {需求 2} | [满足/部分/缺失] | {备注} |
[... 继续所有需求]

### 摘要

**存在的必需章节：** {计数}/{总计}
**合规差距：** {计数}

**严重程度：** [如果缺失监管章节为严重，如果不完整为警告，如果完整为通过]

**建议：**
[如果严重] "PRD 缺失必需的特定领域合规章节。这些对 {领域} 产品至关重要。"
[如果警告] "一些领域合规章节不完整。加强文档以确保完全合规。"
[如果通过] "所有必需的领域合规章节都存在且文档充分。"
```

### 7. 显示进度并自动继续

显示："**领域合规验证完成**

领域：{领域}（{复杂度}）
合规状态：{状态}

**正在进行下一个验证检查...**"

立即完整阅读并遵循：{nextStepFile}（step-v-09-project-type-validation.md）

## 系统成功/失败指标

### 成功：

- 领域分类提取正确
- 复杂度评估适当
- 低复杂度领域：跳过并有清晰的"不适用"文档
- 高复杂度领域：所有必需章节已检查
- 合规矩阵已构建，每个需求都有状态
- 严重程度已正确评估
- 发现已报告到验证报告
- 自动进入下一个验证步骤
- 已尝试子流程并优雅降级

### 失败：

- 在继续之前未检查领域分类
- 对低复杂度领域执行详细检查
- 对于高复杂度：缺失必需章节检查
- 未构建合规矩阵
- 未向验证报告报告发现
- 未自动继续

**主规则：** 领域合规是条件性的。高复杂度领域需要特殊章节 - 低复杂度领域跳过这些检查。
