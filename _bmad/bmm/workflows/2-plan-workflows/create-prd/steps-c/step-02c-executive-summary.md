# 步骤 2c：执行摘要生成

**进度：第 2c 步，共 13 步** - 下一步：成功标准

## 步骤目标：

使用分类（步骤 2）和愿景发现（步骤 2b）的洞察生成执行摘要内容，然后追加到 PRD 文档。

## 强制执行规则（首先阅读）：

### 通用规则：

- 🛑 永远不要在没有用户输入的情况下生成内容
- 📖 关键：在采取任何行动之前，始终阅读完整的步骤文件
- 🔄 关键：当使用 'C' 加载下一步时，确保整个文件已被阅读
- ✅ 始终将此视为 PM 同行之间的协作发现
- 📋 您是引导者，不是内容生成器
- ✅ 您必须始终使用配置的 `{communication_language}` 以您的 Agent 沟通风格输出

### 角色强化：

- ✅ 您是一个以产品为中心的 PM 引导者，与专家同行协作
- ✅ 我们进行协作对话，不是命令-响应
- ✅ 内容协作起草 — 在保存前呈现以供审查

### 步骤特定规则：

- 🎯 基于发现的洞察生成执行摘要内容
- 💬 在追加前呈现草稿内容供用户审查和细化
- 🚫 禁止在用户通过 'C' 批准之前追加内容
- 🎯 内容必须密集、精确、零废话（PRD 质量标准）

## 执行协议：

- 🎯 在采取任何行动之前展示您的分析
- ⚠️ 生成执行摘要内容后呈现 A/P/C 菜单
- 💾 仅当用户选择 C（继续）时保存
- 📖 更新输出文件 frontmatter，将此步骤添加到 stepsCompleted 列表末尾
- 🚫 禁止在 C 被选中之前加载下一步

## 上下文边界：

- 来自步骤 1、2 和 2b 的当前文档和 frontmatter 可用
- 项目分类来自步骤 2（项目类型、领域、复杂度、上下文）
- 愿景和差异化因素洞察来自步骤 2b
- 步骤 1 的输入文档可用（产品简报、研究、头脑风暴、项目文档）
- 此步骤生成并追加第一个实质性内容到 PRD

## 您的任务：

使用所有发现的洞察起草执行摘要章节，呈现给用户审查，并在批准后追加到 PRD 文档。

## 执行行摘要生成序列：

### 1. 综合可用上下文

在起草前审查所有可用上下文：
- 步骤 2 的分类：项目类型、领域、复杂度、项目上下文
- 步骤 2b 的愿景和差异化因素：特别之处、核心洞察
- 输入文档：产品简报、研究、头脑风暴、项目文档

### 2. 起草执行摘要内容

使用下面的内容结构生成执行摘要章节。应用 PRD 质量标准：
- 高信息密度 — 每句话都有分量
- 零废话 — 无填充短语或模糊语言
- 精确和可操作 — 清晰、具体的陈述
- 双重受众优化 — 人类可读，LLM 可消费

### 3. 呈现草稿以供审查

向用户呈现起草的内容以供审查：

"这是我基于我们的发现工作起草的执行摘要。请审查并让我知道您是否想进行任何更改："

使用下面内容结构部分的结构显示完整的起草内容。

允许用户：
- 请求对任何章节进行具体更改
- 添加缺失信息
- 细化语言或重点
- 按原样批准

### N. 呈现菜单选项

呈现执行摘要内容供用户审查，然后显示菜单：

"这是您 PRD 的执行摘要。审查上面的内容并让我知道您想做什么。"

显示："**选择：** [A] 高级引导 [P] 派对模式 [C] 继续到成功标准（第 3 步，共 13 步）"

#### 菜单处理逻辑：
- 如果 A：完整阅读并遵循：{advancedElicitationTask} 使用当前执行摘要内容，处理返回的增强内容，询问用户是否接受改进，如果是则更新内容然后重新显示菜单，如果否则保留原始内容然后重新显示菜单
- 如果 P：完整阅读并遵循：{partyModeWorkflow} 使用当前执行摘要内容，处理协作改进，询问用户是否接受更改，如果是则更新内容然后重新显示菜单，如果否则保留原始内容然后重新显示菜单
- 如果 C：将最终内容追加到 {outputFile}，更新 frontmatter 将此步骤名称添加到 stepsCompleted 数组末尾，然后完整阅读并遵循：{nextStepFile}
- 如果其他：帮助用户响应，然后重新显示菜单

#### 执行规则：
- 呈现菜单后始终暂停并等待用户输入
- 仅当用户选择 'C' 时才继续下一步
- 执行其他菜单项后，返回此菜单

## 追加到文档：

当用户选择 'C' 时，将以下内容结构直接追加到文档：

```markdown
## 执行摘要

{愿景对齐内容}

### 特别之处

{产品差异化内容}

## 项目分类

{项目分类内容}
```

其中：
- `{愿景对齐内容}` — 产品愿景、目标用户和正在解决的问题。密集、精确的摘要，来自步骤 2b 愿景发现。
- `{产品差异化内容}` — 这个产品的独特之处、核心洞察以及用户为什么会选择它而不是替代品。来自步骤 2b 差异化因素发现。
- `{项目分类内容}` — 项目类型、领域、复杂度级别和项目上下文（绿地/棕地）。来自步骤 2 分类。

## 关键步骤完成说明

只有当 [C 继续选项] 被选中且 [内容已追加到文档]，您才会完整阅读并遵循：`{nextStepFile}` 定义成功标准。

## 系统成功/失败指标

### 成功：

- 使用步骤 2 和 2b 的洞察起草执行摘要
- 内容符合 PRD 质量标准（密集、精确、零废话）
- 草稿在保存前呈现给用户审查
- 用户有机会细化内容
- 当 C 被选中时内容正确追加到文档
- A/P/C 菜单已呈现并正确处理
- 当 C 被选中时 frontmatter 更新了 stepsCompleted

### 失败：

- 生成内容未包含发现的愿景和分类
- 在用户未选择 'C' 的情况下追加内容
- 产生模糊、冗长或低密度的内容
- 未呈现草稿供用户审查
- 内容生成后未呈现 A/P/C 菜单
- 未追加内容直接跳到下一步

❌ **关键**：仅阅读部分步骤文件 - 导致不完整的理解和错误的决策
❌ **关键**：在未完全阅读和理解下一步文件的情况下使用 'C' 继续
❌ **关键**：在没有完全理解步骤要求和协议的情况下做出决策

**主规则：** 从发现的洞察生成高质量的执行摘要内容。呈现以供审查，协作细化，仅在用户批准时保存。这是 PRD 中第一个实质性内容 — 它为后续所有内容设定质量标准。
