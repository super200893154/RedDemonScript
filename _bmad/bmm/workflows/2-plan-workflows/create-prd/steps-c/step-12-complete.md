- 所有协作内容已生成、保存和优化

## 您的任务：

完成 PRD 工作流，更新状态文件，提供验证选项，并建议项目的下一步。

## 工作流完成序列：

### 1. 宣布工作流完成

通知用户 PRD 已完成和润色：
- 庆祝全面 PRD 的成功完成
- 总结创建的所有章节
- 突出文档已为流程和连贯性进行了润色
- 强调文档已准备好进行下游工作

### 2. 工作流状态更新

如有主工作流状态文件则更新：

- 从工作流配置加载 `{status_file}`（如存在）
- 更新 workflow_status["prd"] = "{default_output_file}"
- 保存文件，保留所有注释和结构
- 将当前时间戳标记为完成时间

### 3. 验证工作流选项

提供验证工作流以确保 PRD 准备好实施：

**可用验证工作流：**

**选项 1：检查实施就绪度**（`{checkImplementationReadinessWorkflow}`）
- 验证 PRD 具有开发所需的所有信息
- 检查 epic 覆盖完整性
- 审查 UX 与需求的一致性
- 评估 epic 质量和就绪度
- 在架构/设计工作开始前识别差距

**何时使用：** 在开始技术架构或 epic 分解之前

**选项 2：暂时跳过**
- 直接进入下一个工作流（架构、UX、epic）
- 如需要可稍后进行验证
- 一些团队更喜欢在架构审查期间进行验证

### 4. 建议下一个工作流

PRD 完成。完整阅读并遵循：`{project-root}/_bmad/core/tasks/help.md`

### 5. 最终完成确认

- 与用户确认完成并总结已完成的工作
- 文档现在包含：执行摘要、成功标准、用户旅程、领域需求（如适用）、创新分析（如适用）、项目类型需求、功能需求（能力契约）、非功能需求，并已为流程和连贯性进行了润色
- 询问他们是否想运行验证工作流或进入下一个工作流

## 成功指标：

✅ PRD 文档包含所有必需章节并已润色
✅ 所有协作内容正确保存和优化
✅ 工作流状态文件已用完成信息更新（如存在）
✅ 验证工作流选项清晰呈现
✅ 向用户提供清晰的下一步指导
✅ 文档质量验证已完成
✅ 用户确认完成并理解下一步选项

## 失败模式：

❌ 未用完成信息更新工作流状态文件（如存在）
❌ 未提供验证工作流选项
❌ 缺少对用户的清晰下一步指导
❌ 未与用户确认文档完整性
❌ 工作流未在状态跟踪中正确标记为完成（如适用）
❌ 用户不清楚接下来会发生什么或存在哪些验证选项

❌ **关键**：仅阅读部分步骤文件 - 导致不完整的理解和错误的决策
❌ **关键**：在没有完全理解步骤要求和协议的情况下做出决策
