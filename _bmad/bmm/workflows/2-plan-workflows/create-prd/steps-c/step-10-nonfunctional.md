## 非功能需求序列：

### 1. 解释 NFR 目的和范围

首先阐明什么是 NFR 以及我们为什么选择性：

**NFR 目的：**
NFR 定义系统必须执行得多好，而不是必须做什么。它们指定性能、安全性、可扩展性等质量属性。

**选择性方法：**
我们只记录对这个产品重要的 NFR。如果某个类别不适用，我们完全跳过它。这防止需求膨胀并专注于实际重要的内容。

### 2. 评估产品上下文的 NFR 相关性

根据产品上下文评估哪些 NFR 类别重要：

**快速评估问题：**

- **性能**：速度是否有面向用户的影响？
- **安全性**：我们是否处理敏感数据或支付？
- **可扩展性**：我们是否预期用户快速增长？
- **可访问性**：我们是否服务于广泛的公众受众？
- **集成**：我们是否需要与其他系统连接？
- **可靠性**：停机会导致重大问题吗？

### 3. 探索相关的 NFR 类别

对于每个相关类别，进行有针对性的发现：

#### 性能 NFR（如果相关）：

探索性能需求：
- 系统的哪些部分需要快速才能让用户成功？
- 是否有特定的响应时间期望？
- 如果性能比预期慢会发生什么？
- 是否有我们需要支持的并发用户场景？

#### 安全性 NFR（如果相关）：

探索安全需求：
- 哪些数据需要保护？
- 谁应该有权访问什么？
- 我们需要缓解哪些安全风险？
- 是否有合规要求（GDPR、HIPAA、PCI-DSS）？

#### 可扩展性 NFR（如果相关）：

探索可扩展性需求：
- 我们最初预期多少用户？长期呢？
- 是否有季节性或基于事件的流量高峰？
- 如果超出容量会发生什么？
- 我们应该为什么增长场景做计划？

#### 可访问性 NFR（如果相关）：

探索可访问性需求：
- 我们是否服务于有视觉、听觉或运动障碍的用户？
- 是否有法律可访问性要求（WCAG、Section 508）？
- 哪些可访问性功能对我们的用户最重要？

#### 集成 NFR（如果相关）：

探索集成需求：
- 我们需要连接什么外部系统？
- 是否有我们必须支持的 API 或数据格式？
- 这些集成需要多可靠？

### 4. 使 NFR 具体和可衡量

对于每个相关的 NFR 类别，确保标准可测试：

**从模糊到具体：**

- 不是："系统应该快" → "用户操作在 2 秒内完成"
- 不是："系统应该安全" → "所有数据在静态和传输中加密"
- 不是："系统应该可扩展" → "系统支持 10 倍用户增长，性能下降 <10%"

## NFR 类别指导：

**何时包含性能：**

- 面向用户的响应时间影响成功
- 实时交互至关重要
- 性能是竞争差异化因素

**何时包含安全性：**

- 处理敏感用户数据
- 处理支付或财务信息
- 受合规法规约束
- 保护知识产权

**何时包含可扩展性：**

- 预期用户快速增长
- 处理可变流量模式
- 支持企业规模使用
- 计划市场扩展

**何时包含可访问性：**

- 服务于广泛公众受众
- 受可访问性法规约束
- 目标用户有残疾
- B2B 客户有可访问性要求
