## 您的任务：

对于复杂领域，探索使此领域特别的内容：
- **合规要求** - 法规、标准、认证
- **技术约束** - 安全、隐私、集成要求
- **领域模式** - 常见模式、最佳实践、反模式
- **风险和缓解** - 可能出什么问题，如何预防

## 领域发现序列：

### 1. 检查领域复杂度

**审查步骤 02 的分类：**

- 领域复杂度级别是什么？（低/中/高）
- 具体领域是什么？（医疗保健、金融科技、教育等）

**如果复杂度为低：**

提供跳过：
"我们发现的领域复杂度为低。我们可能不需要深度的领域特定需求。您想：
- [C] 跳过此步骤并进入创新
- [D] 仍然进行领域探索"

**如果复杂度为中或高：**

继续进行领域探索。

### 2. 加载领域参考数据

**尝试子流程数据查找：**

"您的任务：在 {domainComplexityCSV} 中查找数据

**搜索条件：**
- 找到 domain 匹配 {{步骤 02 的领域}} 的行

**返回格式：**
仅返回匹配的行作为 YAML 格式对象，包含以下字段：
domain, complexity, typical_concerns, compliance_requirements

**不要返回整个 CSV - 仅返回匹配的行。**"

**优雅降级（如果 Task 工具不可用）：**
- 直接加载 CSV 文件
- 手动查找匹配行
- 提取所需字段
- 了解典型关注点和合规要求

### 3. 探索领域特定关注点

**从您知道的开始：**

确认领域并探索使其复杂的内容：
- 适用什么法规？（HIPAA、PCI-DSS、GDPR、SOX 等）
- 什么标准重要？（ISO、NIST、特定领域标准）
- 需要什么认证？（安全、隐私、特定领域）
- 需要什么集成？（EMR 系统、支付处理器等）

**探索技术约束：**
- 安全要求（加密、审计日志、访问控制）
- 隐私要求（数据处理、同意、保留）
- 性能要求（实时、批量、延迟）
- 可用性要求（正常运行时间、灾难恢复）

### 4. 记录领域需求

**围绕关键关注点构建需求：**

```markdown
### 合规与监管
- [具体要求]

### 技术约束
- [安全、隐私、性能需求]

### 集成要求
- [所需系统和数据流]

### 风险缓解
- [领域特定风险及如何解决]
```

### 5. 验证完整性

**与用户确认：**

"还有其他我们应该考虑的领域特定关注点吗？对于[此领域]，通常会忽略什么？"
