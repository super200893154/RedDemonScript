## 功能需求综合序列：

### 1. 理解 FR 目的和用法

首先解释功能需求的关键作用：

**目的：**
FR 定义产品必须具备什么能力。它们是交付产品愿景的面向用户和系统能力的完整清单。

**关键属性：**
✅ 每个 FR 是一个可测试的能力
✅ 每个 FR 与实现无关（可以用多种方式构建）
✅ 每个 FR 指定谁和什么，而非如何
✅ 无 UI 细节、无性能数字、无技术选择
✅ 能力领域的全面覆盖

**它们将如何被使用：**

1. UX 设计师阅读 FR → 为每个能力设计交互
2. 架构师阅读 FR → 设计支持每个能力的系统
3. PM 阅读 FR → 创建 epic 和 story 来实现每个能力

### 2. 审查现有内容以提取能力

系统化审查所有先前章节以提取能力：

**从以下提取：**

- 执行摘要 → 核心产品差异化能力
- 成功标准 → 实现成功的能力
- 用户旅程 → 旅程揭示的能力
- 领域需求 → 合规和监管能力
- 创新模式 → 创新功能能力
- 项目类型需求 → 技术能力需求

### 3. 按能力领域组织需求

按逻辑能力领域分组 FR（而非按技术或层次）：

**良好的分组示例：**

- ✅ "用户管理"（不是"认证系统"）
- ✅ "内容发现"（不是"搜索算法"）
- ✅ "团队协作"（不是"WebSocket 基础设施"）

**目标 5-8 个能力领域**用于典型项目。

### 4. 生成全面的 FR 列表

使用以下格式创建完整的功能需求：

**格式：**

- FR#: [参与者] 可以 [能力] [如需要则加上下文/约束]
- 按顺序编号（FR1, FR2, FR3...）
- 典型项目目标为 20-50 个 FR

**高度检查：**
每个 FR 应回答"存在什么能力？"而非"如何实现？"

**示例：**

- ✅ "用户可以自定义外观设置"
- ❌ "用户可以用 3 种字体大小选项切换浅色/深色主题，存储在 LocalStorage 中"

### 5. 自我验证过程

在呈现给用户之前，验证 FR 列表：

**完整性检查：**

1. "我是否覆盖了 MVP 范围章节中提到的每个能力？"
2. "我是否将领域特定需求作为 FR 包含？"
3. "我是否覆盖了项目类型特定需求？"
4. "UX 设计师能否仅阅读 FR 就知道要设计什么？"
5. "架构师能否仅阅读 FR 就知道要支持什么？"
6. "是否有我们讨论过的任何用户操作或系统行为没有 FR？"

**高度检查：**

1. "我在陈述能力（什么）还是实现（如何）？"
2. "我在列出验收标准或 UI 细节吗？"（如果是则删除）
3. "这个 FR 能否以 5 种不同方式实现？"（好 - 意味着它不是规定性的）

**质量检查：**

1. "每个 FR 是否足够清晰以至于有人可以测试它是否存在？"
2. "每个 FR 是否独立（不依赖阅读其他 FR 来理解）？"
3. "我是否避免了'好'、'快'、'易'等模糊术语？"（使用 NFR 处理质量属性）

[继续对话中发现的所有能力领域]
