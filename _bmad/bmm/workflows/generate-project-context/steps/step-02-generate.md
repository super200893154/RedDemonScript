# 步骤 2：上下文规则生成

## 强制执行规则（首先阅读）：

- 🛑 绝不在没有用户输入的情况下生成内容
- ✅ 始终将其视为技术同伴之间的协作发现
- 📋 你是引导者，而非内容生成器
- 💬 专注于 AI agent 需要被提醒的不明显规则
- 🎯 保持内容精简 - 针对 LLM 上下文效率优化
- ⚠️ 绝对不要时间估计 - AI 开发速度已根本改变
- ✅ 你必须始终使用配置中的 `{communication_language}` 以你的 Agent 交流风格输出

## 执行协议：

- 🎯 在采取任何行动之前展示你的分析
- 📝 专注于具体、可执行的规则而非一般建议
- ⚠️ 在每个主要规则类别后呈现 A/P/C 菜单
- 💾 只有当用户选择 C（继续）时才保存
- 📖 用已完成的部分更新 frontmatter
- 🚫 禁止在所有部分完成前加载下一步

## 协作菜单 (A/P/C)：

此步骤将生成内容并为每个规则类别呈现选择：

- **A（高级引导）**：使用发现协议探索细致的实现规则
- **P（Party Mode）**：引入多视角识别关键边界情况
- **C（继续）**：保存当前规则并进入下一类别

## 协议集成：

- 当选择 'A' 时：执行 {project-root}/_bmad/core/workflows/advanced-elicitation/workflow.xml
- 当选择 'P' 时：执行 {project-root}/_bmad/core/workflows/party-mode
- 协议在 A 或 P 完成后总是返回显示此步骤的 A/P/C 菜单
- 用户在继续前接受/拒绝协议更改

## 上下文边界：

- 步骤 1 的发现结果可用
- 技术栈和现有模式已识别
- 专注于防止实现错误的规则
- 优先处理 AI agent 可能遗漏的不明显细节

## 你的任务：

协作生成 AI agent 在此项目实现代码时必须遵循的具体、关键规则。

### 9. 呈现内容和菜单

每个类别后，显示生成的规则并呈现选择：

"我已经为您的项目上下文起草了 {{category_name}} 规则。

**我将添加的内容：**

[显示此类别的完整 markdown 内容]

**您想做什么？**
[A] 高级引导 - 探索此类别的细致规则
[P] Party Mode - 从不同实现视角审查
[C] 继续 - 保存这些规则并进入下一类别"

### 10. 处理菜单选择

#### 如果 'A'（高级引导）：

- 用当前类别规则执行 advanced-elicitation.xml
- 处理返回的增强规则
- 询问用户："接受 {{category}} 的这些增强规则？(y/n)"
- 如果是：更新内容，然后返回 A/P/C 菜单
- 如果否：保留原内容，然后返回 A/P/C 菜单

#### 如果 'P'（Party Mode）：

- 用类别规则上下文执行 party-mode 工作流
- 处理关于实现模式的协作见解
- 询问用户："接受对 {{category}} 规则的这些更改？(y/n)"
- 如果是：更新内容，然后返回 A/P/C 菜单
- 如果否：保留原内容，然后返回 A/P/C 菜单

#### 如果 'C'（继续）：

- 将当前类别内容保存到项目上下文文件
- 更新 frontmatter：`sections_completed: [...]`
- 进入下一类别，或如完成则进入步骤 3

## 追加到项目上下文：

当用户为类别选择 'C' 时，使用步骤 8 的结构将内容直接追加到 `{output_folder}/project-context.md`。
