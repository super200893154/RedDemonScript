# 任务：BMAD 帮助

## 路由规则

- **空的 `phase` = 任何时候** — 通用工具无论工作流状态如何都可用
- **编号阶段表示顺序** — 阶段如 `1-discover` → `2-define` → `3-build` → `4-ship` 按顺序流转（命名因模块而异）
- **无必需步骤的阶段** — 如果整个阶段没有必需的、为真的项目，则整个阶段是可选的。如果它在另一个阶段之前按顺序排列，可以推荐，但始终向用户明确说明真正的下一个必需项目是什么。
- **保持在模块内** — 根据阶段+序列排序引导完成活动模块的工作流
- **描述包含路由信息** — 阅读以了解替代路径（例如，"如果需要修复则返回上一阶段"）
- **`required=true` 阻塞进度** — 必需的工作流必须完成后才能进入后续阶段
- **工件揭示完成状态** — 搜索已解析的输出路径中的 `outputs` 模式，将找到的文件与工作流行进行模糊匹配

## 显示规则

### 基于命令的工作流
当 `command` 字段有值时：
- 显示以 `/` 为前缀的命令（例如 `/bmad-bmm-create-prd`）

### 基于代理的工作流
当 `command` 字段为空时：
- 用户首先通过 `/agent-command` 加载代理
- 然后通过引用 `code` 字段或描述 `name` 字段来调用
- 不要显示斜杠命令 — 而是显示代码值和代理加载指令

空命令的示例展示：
```
解释概念 (EC)
加载：/tech-writer，然后请求"EC关于[主题]"
代理：技术文档撰写员
描述：创建带有示例的清晰技术解释...
```

## 模块检测

- **空的 `module` 列** → 通用工具（跨所有模块工作）
- **命名的 `module`** → 模块特定的工作流

从对话上下文、最近的工作流或用户查询关键词中检测活动模块。如果模糊不清，询问用户。

## 输入分析

确定刚刚完成的内容：
- 用户明确声明的完成
- 当前对话中完成的工作流
- 找到匹配 `outputs` 模式的工件
- 如果 `index.md` 存在，读取它以获取额外上下文
- 如果仍然不清楚，询问："您最近完成了哪个工作流？"

## 执行

1. **加载目录** — 加载 `{project-root}/_bmad/_config/bmad-help.csv`

2. **解析输出位置和配置** — 扫描 `{project-root}/_bmad/` 下的每个文件夹（除 `_config` 外）查找 `config.yaml`。对于每个工作流行，根据该模块的配置解析其 `output-location` 变量，以便可以搜索工件路径。同时从每个扫描模块的配置中提取 `communication_language` 和 `project_knowledge`。

3. **基于项目知识** — 如果 `project_knowledge` 解析为现有路径，阅读可用的文档文件（架构文档、项目概述、技术栈参考）以获取基础上下文。在编写任何项目特定输出时使用发现的项目事实。永远不要编造项目特定细节 — 如果文档不可用，请说明。

4. **检测活动模块** — 使用上面的模块检测

5. **分析输入** — 任务可能提供工作流名称/代码、对话短语或什么都不提供。使用上面的输入分析推断刚刚完成的内容。

6. **呈现建议** — 基于以下内容显示下一步：
   - 检测到的已完成工作流
   - 阶段/序列排序（路由规则）
   - 工件存在

   **可选项目优先** — 列出可选工作流直到遇到必需步骤
   **必需项目其次** — 列出下一个必需工作流

   对于每个项目，应用上面的显示规则并包括：
   - 工作流**名称**
   - **命令** 或 **代码 + 代理加载指令**（根据显示规则）
   - 来自CSV的**代理**标题和显示名称（例如 "🎨 Alex (设计师)"）
   - 简短**描述**

7. **需要传达的额外指导**：
   - 以 `{communication_language}` 呈现所有输出
   - 在**新的上下文窗口**中运行每个工作流
   - 对于**验证工作流**：如有条件推荐使用不同的高质量LLM
   - 对于对话请求：在清晰呈现的同时匹配用户的语气

8. 呈现建议后返回调用进程。
