# BMAD 技术文档标准

CommonMark 标准、技术写作最佳实践和风格指南合规性。

## 用户指定的关键规则 - 优先于一般关键规则

无

## 一般关键规则

### 规则 1：CommonMark 严格合规

所有文档必须严格遵循 CommonMark 规范。无例外。

### 规则 2：禁止时间估计

绝不记录任何工作流、任务或活动的时间估计、持续时间、工作量或完成时间，除非用户明确要求。这包括：

- 禁止工作流执行时间（例如 "30-60 分钟"、"2-8 小时"）
- 禁止任务持续时间和工作量估计
- 禁止阅读时间估计
- 禁止实现时间范围
- 禁止任何基于时间或容量的测量

**替代方案**：专注于工作流步骤、依赖项和输出。让用户自行确定时间线和工作量。

### CommonMark 要点

**标题：**

- 仅使用 ATX 风格：`#` `##` `###`（不使用 Setext 下划线）
- `#` 后单空格：`# 标题`（不是 `#标题`）
- 无尾部 `#`：`# 标题`（不是 `# 标题 #`）
- 层级顺序：不跳级（h1→h2→h3，不是 h1→h3）

**代码块：**

- 使用带语言标识符的围栏块：
  ````markdown
  ```javascript
  const example = 'code';
  ```
  ````
- 不使用缩进代码块（有歧义）

**列表：**

- 列表内标记一致：全部使用 `-` 或全部使用 `*` 或全部使用 `+`（不要混用）
- 嵌套项正确缩进（2 或 4 个空格，保持一致）
- 列表前后空行以保持清晰

**链接：**

- 内联：`[文本](url)`
- 引用：`[文本][ref]` 然后在底部 `[ref]: url`
- 无 `<>` 括号的裸 URL 不使用

**强调：**

- 斜体：`*文本*` 或 `_文本_`
- 粗体：`**文本**` 或 `__文本__`
- 文档内风格一致

**换行：**

- 行尾两个空格 + 换行，或
- 段落之间空行
- 不使用单换行（它们被忽略）

## Mermaid 图表：需要有效语法

**关键规则：**

1. 第一行始终指定图表类型
2. 使用有效的 Mermaid v10+ 语法
3. 输出前测试语法（心理验证）
4. 保持专注：理想 5-10 个节点，最多 15 个

**图表类型选择：**

- **flowchart** - 流程流、决策树、工作流
- **sequenceDiagram** - API 交互、消息流、基于时间的过程
- **classDiagram** - 对象模型、类关系、系统结构
- **erDiagram** - 数据库模式、实体关系
- **stateDiagram-v2** - 状态机、生命周期阶段
- **gitGraph** - 分支策略、版本控制流

**格式化：**

````markdown
```mermaid
flowchart TD
    Start[清晰标签] --> Decision{问题?}
    Decision -->|是| Action1[执行此操作]
    Decision -->|否| Action2[执行彼操作]
```
````

## 风格指南原则（精简版）

按此层次应用：

1. **项目特定指南**（如存在）- 始终先询问
2. **BMAD 约定**（本文档）
3. **Google 开发者文档风格**（以下为默认值）
4. **CommonMark 规范**（如有疑问）

### 核心写作规则

**任务导向重点：**

- 为用户目标而写，而非功能列表
- 从为什么开始，然后是如何
- 每个文档回答："我能完成什么？"

**清晰原则：**

- 主动语态："点击按钮" 不是 "按钮应该被点击"
- 现在时态："函数返回" 不是 "函数将返回"
- 直接语言："使用 X 来做 Y" 不是 "X 可以被用来做 Y"
- 第二人称："您配置" 不是 "用户配置" 或 "一个人配置"

**结构：**

- 一句话一个观点
- 一段一个主题
- 标题准确描述内容
- 示例紧随解释之后

**可访问性：**

- 描述性链接文本："查看 API 参考" 不是 "点击这里"
- 图表的替代文本：描述它显示的内容
- 语义标题层次（不跳级）
- 表格有标题

## OpenAPI/API 文档

**必需元素：**

- 端点路径和方法
- 认证要求
- 请求参数（路径、查询、正文）及类型
- 请求示例（真实、可用）
- 响应模式及类型
- 响应示例（成功 + 常见错误）
- 错误代码和含义

**质量标准：**

- OpenAPI 3.0+ 规范合规
- 完整模式（无缺失字段）
- 实际可用的示例
- 清晰的错误消息
- 已记录安全方案

## 文档类型：快速参考

**README：**

- 是什么（概述）、为什么（目的）、怎么做（快速入门）
- 安装、使用、贡献、许可证
- 500 行以内（链接到详细文档）
- 最后润色包含目录

**API 参考：**

- 完整端点覆盖
- 请求/响应示例
- 认证详情
- 错误处理
- 如适用，限流

**用户指南：**

- 基于任务的章节（如何...）
- 分步说明
- 截图/图表（如有帮助）
- 故障排除部分

**架构文档：**

- 系统概述图（Mermaid）
- 组件描述
- 数据流
- 技术决策（ADR）
- 部署架构

**开发者指南：**

- 设置/环境要求
- 代码组织
- 开发工作流
- 测试方法
- 贡献指南

## 质量清单

在完成任何文档之前：

- [ ] CommonMark 合规（无违规）
- [ ] 无任何时间估计（关键规则 2）
- [ ] 标题层级正确
- [ ] 所有代码块有语言标签
- [ ] 链接有效且有描述性文本
- [ ] Mermaid 图表正确渲染
- [ ] 主动语态、现在时态
- [ ] 任务导向（回答"我如何..."）
- [ ] 示例具体可用
- [ ] 满足可访问性标准
- [ ] 拼写/语法已检查
- [ ] 在目标技能水平上阅读清晰
