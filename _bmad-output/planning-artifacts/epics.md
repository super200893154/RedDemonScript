# RedDemonScript - Epic 分解

## 概览

本文档提供 RedDemonScript 的完整 epic 和 story 分解，将 PRD、UX 设计和架构需求中的需求分解为可实施的 story。

## 需求清单

### 功能需求

**核心脚本执行（6 个）：**
- FR1: 用户可以通过实现 IGameLauncher 接口定义游戏启动逻辑
- FR2: 用户可以通过实现 IDungeonEntry 接口定义副本入口逻辑
- FR3: 用户可以通过实现 IDungeonRunner 接口定义副本战斗逻辑
- FR4: 系统可以循环执行刷图流程直到收到停止指令
- FR5: 系统可以调用外部 DLL 进行图色识别操作
- FR6: 系统可以调用外部 DLL 进行键鼠模拟操作

**异常监控与恢复（4 个）：**
- FR7: 系统可以检测游戏进程是否无响应
- FR8: 系统可以检测游戏进程是否意外终止（闪退）
- FR9: 系统可以在检测到异常后自动重启游戏并恢复运行
- FR10: 系统可以在异常恢复时记录异常详情到日志

**定时器管理（2 个）：**
- FR11: 系统可以在配置的时间间隔后触发定时重启
- FR12: 系统可以在定时重启时间上添加随机偏移（模拟真人行为）

**多账号管理（8 个）：**
- FR13: 用户可以添加、编辑、删除账号信息
- FR14: 用户可以配置账号的最大运行时间
- FR15: 用户可以配置账号的轮换顺序
- FR16: 用户可以启用或禁用特定账号
- FR17: 系统可以按配置顺序自动轮换账号
- FR18: 系统可以在 Sandboxie 沙箱中启动游戏实例
- FR19: 系统可以在账号切换时终止当前沙箱中的所有进程
- FR20: 系统可以为每个账号维护独立的日志存储

**图形界面（5 个）：**
- FR21: 用户可以通过主窗口启动脚本执行
- FR22: 用户可以通过主窗口停止脚本执行
- FR23: 用户可以查看当前运行状态（运行中/已停止/异常）
- FR24: 用户可以查看账号列表及其状态
- FR25: 用户可以查看当前运行的账号信息

**日志系统（5 个）：**
- FR26: 系统可以记录带时间戳的操作日志
- FR27: 系统可以记录异常事件详情
- FR28: 系统可以按账号隔离存储日志
- FR29: 用户可以查看当前账号的运行日志
- FR30: 系统可以自动清理超过 7 天的日志文件

**配置管理（5 个）：**
- FR31: 系统可以从 JSON 文件加载配置
- FR32: 系统可以将配置保存到 JSON 文件
- FR33: 用户可以配置定时重启的时间间隔
- FR34: 用户可以配置 Sandboxie 安装路径
- FR35: 用户可以配置游戏进程名称

**功能需求总计：35 个**

### 非功能需求

**可靠性（5 个）：**
- NFR1: 系统可以连续运行 ≥ 12 小时不崩溃
- NFR2: 系统内存占用在长时间运行中保持稳定（无内存泄漏）
- NFR3: 三线程协调运行无死锁、无竞态条件
- NFR4: 异常恢复成功率 ≥ 90%
- NFR5: 系统可以在游戏闪退后 60 秒内完成恢复

**性能（4 个）：**
- NFR6: 系统内存占用 < 100MB（不含游戏进程）
- NFR7: 系统CPU占用 < 5%（不含游戏进程和DLL操作）
- NFR8: UI 操作响应时间 < 200ms
- NFR9: 日志写入延迟 < 50ms

**集成（3 个）：**
- NFR10: 系统可以与 Sandboxie Plus 5.55+ 版本兼容
- NFR11: 外部 DLL 可替换，无需重新编译主程序
- NFR12: 配置文件格式向后兼容（v1.x 配置可在 v1.y 中使用）

**可维护性（3 个）：**
- NFR13: 代码模块化，用户接口与框架代码分离
- NFR14: 日志格式标准化，支持问题快速定位
- NFR15: 配置文件采用 JSON 格式，人类可读可编辑

**非功能需求总计：15 个**

## Epic 列表

### Epic 1: 项目基础设施与配置

**用户成果：** 用户可以配置应用程序，配置被持久化保存，日志系统正常运行。这是所有后续功能的基础。

**覆盖的 FR：** FR26, FR28, FR30-FR35（配置管理 5 个 + 日志基础 4 个）

**用户成果：** 用户可以实现自己的游戏逻辑（启动/进入副本/战斗），并执行自动化脚本循环。

**覆盖的 FR：** FR1-FR6（核心脚本执行 6 个）

---

### Epic 3: 异常监控与恢复

**用户成果：** 用户可以安心挂机，系统自动检测游戏异常（闪退/无响应）并恢复运行。

### Epic 4: 定时器管理

**用户成果：** 系统可以定时重启游戏，带有随机偏移模拟真人行为，预防性维护稳定性。

### Epic 5: 多账号管理

**用户成果：** 用户可以管理多个游戏账号，配置轮换顺序，系统自动按计划切换账号运行。

### Epic 6: 图形界面

## Epic 1: 项目基础设施与配置

**用户成果：** 用户可以配置应用程序，配置被持久化保存，日志系统正常运行。这是所有后续功能的基础。

**覆盖的 FR：** FR26, FR28, FR30-FR35

---

### Story 1.1: 项目结构搭建

作为一个 **开发者**，
我想要 **建立完整的项目目录结构和构建配置**，
以便 **后续开发有清晰的代码组织基础**。

**验收标准：**

**假设** 项目已有基础的 Qt Creator 模板文件
**当** 开发者执行构建配置
**那么** 项目目录结构符合架构设计
**并且** CMakeLists.txt 配置 C++17 和所需 Qt 组件
**并且** 所有目录创建完成
**并且** 项目可以成功编译运行空白窗口

**技术要点：**
- 扩展 CMakeLists.txt：添加 QuickControls2、Widgets、Concurrent 组件
- 设置 CMAKE_CXX_STANDARD 17
- 创建目录：src/core/, src/interfaces/, src/helpers/, src/account/, src/ui/, qml/components/, config/
- 配置 Material Dark 主题（qtquickcontrols2.conf）

---

### Story 1.2: 日志系统实现

作为一个 **系统**，
我想要 **记录带时间戳的操作日志并按账号隔离存储**，
以便 **用户可以追溯问题和了解运行状态**。

**验收标准：**

**假设** 项目结构已搭建完成
**当** 系统需要记录日志
**那么** 日志包含时间戳、级别、账号名和操作描述
**并且** 日志按账号隔离存储到独立目录
**并且** 日志写入延迟 < 50ms（NFR9）
**并且** 日志格式：`[YYYY-MM-DD HH:mm:ss] [LEVEL] 账号名 - 操作描述`

**覆盖 FR：** FR26, FR28

---

### Story 1.3: 日志自动清理

作为一个 **系统**，
我想要 **自动清理超过 7 天的日志文件**，
以便 **磁盘空间不会被历史日志占满**。

**验收标准：**

**假设** 日志系统已实现
**当** 应用启动时
**那么** 系统检查日志目录
**并且** 删除超过 7 天的日志文件
**并且** 记录清理操作到日志

**覆盖 FR：** FR30

---

### Story 1.4: 配置管理器实现

作为一个 **系统**，
我想要 **从 JSON 文件加载配置并将配置保存到 JSON 文件**，
以便 **用户配置可以在应用重启后保留**。

**验收标准：**

**假设** 日志系统已实现
**当** 应用启动时
**那么** 系统从 config/config.json 加载配置
**当** 配置变更时
**那么** 系统将配置保存到 config/config.json
**并且** 配置文件格式为 JSON，人类可读（NFR15）
**并且** 配置包含：定时重启间隔、Sandboxie 路径、游戏进程名、账号列表

**覆盖 FR：** FR31, FR32, FR33, FR34, FR35

---

### Story 1.5: 账号密码加密

作为一个 **用户**，
我想要 **账号密码被安全加密存储**，
以便 **我的游戏账号信息安全**。

**验收标准：**

**假设** 配置管理器已实现
**当** 用户保存账号信息
**那么** 密码使用 Windows DPAPI 加密
**当** 用户读取账号信息
**那么** 密码自动解密显示
**并且** 加密后的密码存储在配置文件中

**技术要点：**
- 使用 Windows DPAPI（CryptProtectData/CryptUnprotectData）
- 仅在 Windows 平台可用

## Epic 2: 核心脚本执行

**用户成果：** 用户可以实现自己的游戏逻辑（启动/进入副本/战斗），并执行自动化脚本循环。

**覆盖的 FR：** FR1-FR6

---

### Story 2.1: 用户接口定义

作为一个 **开发者（用户）**，
我想要 **定义游戏启动、副本入口、副本战斗三个接口**，
以便 **我可以实现自己的游戏逻辑**。

**验收标准：**

**假设** 项目结构已搭建完成
**当** 开发者查看 src/interfaces/ 目录
**那么** 存在 IGameLauncher.h、IDungeonEntry.h、IDungeonRunner.h 三个接口文件
**并且** 每个接口定义纯虚函数供用户实现
**并且** 接口命名遵循 I 前缀约定

**接口定义：**
- IGameLauncher: `virtual bool launch(const QString& account) = 0;`
- IDungeonEntry: `virtual bool enterDungeon() = 0;`
- IDungeonRunner: `virtual bool runDungeon() = 0;`

**覆盖 FR：** FR1, FR2, FR3

---

### Story 2.2: 外部 DLL 封装

作为一个 **系统**，
我想要 **通过封装层调用外部 DLL 进行图色识别和键鼠模拟**，
以便 **用户可以替换底层实现而无需重新编译主程序**。

**验收标准：**

**假设** 用户接口已定义
**当** 系统需要执行图色识别或键鼠模拟
**那么** 通过 ImageHelper 和 InputHelper 封装类调用
**并且** 使用 QLibrary 动态加载 DLL
**并且** DLL 可替换，无需重新编译主程序（NFR11）
**并且** 封装类提供友好的 Qt 风格 API

**封装类：**
- ImageHelper: findImage(), waitImage(), getColor()
- InputHelper: click(), move(), pressKey()

**覆盖 FR：** FR5, FR6

---

### Story 2.3: ScriptController 主控制器

作为一个 **系统**，
我想要 **一个主控制器协调脚本执行、状态管理和线程通信**，
以便 **各模块可以统一协作**。

**验收标准：**

**假设** 外部 DLL 封装已实现
**当** 系统运行时
**那么** ScriptController 作为单例管理全局状态
**并且** 提供 Q_PROPERTY 供 QML 绑定
**并且** 通过信号槽与工作线程通信
**并且** 状态变化自动通知 UI 更新

**状态属性：**
- isRunning: bool
- currentAccount: QString
- runTime: qint64
- status: enum (Running/Stopped/Error)

**覆盖架构需求：** 状态管理模式

---

### Story 2.4: ScriptThread 脚本执行线程

作为一个 **系统**，
我想要 **在独立线程中循环执行刷图流程**，
以便 **主线程保持响应，脚本稳定运行**。

**验收标准：**

**假设** ScriptController 已实现
**当** 用户启动脚本
**那么** ScriptThread 开始执行刷图循环
**当** 用户停止脚本
**那么** ScriptThread 停止循环并退出
**并且** 循环流程：启动游戏 → 进入副本 → 战斗 → 重复
**并且** 通过信号槽通知 ScriptController 状态变化
**并且** 使用 Qt::QueuedConnection 跨线程通信

**覆盖 FR：** FR4

## Epic 3: 异常监控与恢复

**用户成果：** 用户可以安心挂机，系统自动检测游戏异常（闪退/无响应）并恢复运行。

**覆盖的 FR：** FR7-FR10, FR27

---

### Story 3.1: MonitorThread 异常监控线程

作为一个 **系统**，
我想要 **一个独立的监控线程持续检测游戏进程状态**，
以便 **异常可以被及时发现并处理**。

**验收标准：**

**假设** ScriptController 已实现
**当** 脚本开始运行
**那么** MonitorThread 启动并开始监控
**当** 脚本停止运行
**那么** MonitorThread 停止监控并退出
**并且** 监控间隔可配置（默认 5 秒）
**并且** 通过信号槽通知 ScriptController 异常事件

---

### Story 3.2: 进程无响应检测

作为一个 **系统**，
我想要 **检测游戏进程是否无响应**，
以便 **可以在游戏卡死时自动恢复**。

**验收标准：**

**假设** MonitorThread 已实现
**当** 监控线程检测游戏进程
**那么** 使用 Windows API 检测进程响应状态
**当** 进程无响应超过阈值（默认 30 秒）
**那么** 发送 `processNotResponding` 信号
**并且** 记录 WARN 级别日志

**技术要点：**
- 使用 IsHungAppWindow 或发送 WM_NULL 消息检测

**覆盖 FR：** FR7

---

### Story 3.3: 进程闪退检测

作为一个 **系统**，
我想要 **检测游戏进程是否意外终止（闪退）**，
以便 **可以在游戏崩溃时自动恢复**。

**验收标准：**

**假设** MonitorThread 已实现
**当** 监控线程检测游戏进程
**那么** 检查进程 ID 是否仍存在
**当** 进程已终止且非正常退出
**那么** 发送 `processCrashed` 信号
**并且** 记录 ERROR 级别日志

**覆盖 FR：** FR8

---

### Story 3.4: 自动恢复机制

作为一个 **用户**，
我想要 **系统在检测到异常后自动重启游戏并恢复运行**，
以便 **我可以安心睡眠，无需担心脚本中断**。

**验收标准：**

**假设** 进程检测功能已实现
**当** 系统检测到进程无响应或闪退
**那么** 终止残留进程（如有）
**当** 终止完成后
**那么** 重新启动游戏
**并且** 自动恢复到刷图流程
